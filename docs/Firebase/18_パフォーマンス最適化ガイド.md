## 1. パフォーマンス計測の基本

改善の前に必ず計測します。「計測なき最適化」は時間の無駄になります。

### 計測すべき指標

| 指標 | 説明 | 目標値 |
|------|------|-------|
| **LCP**（Largest Contentful Paint） | 最大コンテンツの表示時間 | < 2.5 秒 |
| **FID/INP**（インタラクションの遅延） | ユーザー操作への応答時間 | < 200 ms |
| **CLS**（レイアウトシフト） | 画面のガタつき | < 0.1 |
| **Firestore 読み取り数** | 月間コストに直結 | 無料枠 50,000 回/日以内 |
| **Functions 実行時間** | タイムアウト・コスト | < 10 秒 |

### 計測ツール

```bash
# Chrome DevTools → Network タブ でウォーターフォール確認
# Lighthouse（Chrome DevTools → Lighthouse タブ）

# Firebase コンソール → Performance Monitoring でリアルユーザー計測
```

## 2. Firestore のクエリ最適化

### ❌ N+1 問題：ループ内で個別取得

```typescript
// NG: 10件の投稿があれば、著者情報の取得で 10 回の読み取りが発生
const postsSnapshot = await getDocs(collection(db, 'posts'));
const posts = await Promise.all(
  postsSnapshot.docs.map(async (postDoc) => {
    const authorDoc = await getDoc(doc(db, 'users', postDoc.data().authorId));
    return { ...postDoc.data(), author: authorDoc.data() };
  })
);
```

```typescript
// ✅ 著者 ID を収集して一括取得（in クエリは 30件まで）
const postsSnapshot = await getDocs(collection(db, 'posts'));
const authorIds = [...new Set(postsSnapshot.docs.map((d) => d.data().authorId))];

// 30件を超える場合はチャンク分割
function chunkArray<T>(arr: T[], size: number): T[][] {
  return Array.from({ length: Math.ceil(arr.length / size) }, (_, i) =>
    arr.slice(i * size, i * size + size)
  );
}

const authorChunks = chunkArray(authorIds, 30);
const authorSnapshots = await Promise.all(
  authorChunks.map((chunk) =>
    getDocs(query(collection(db, 'users'), where('__name__', 'in', chunk)))
  )
);

const authorMap = new Map(
  authorSnapshots.flatMap((snap) =>
    snap.docs.map((d) => [d.id, d.data()])
  )
);

const posts = postsSnapshot.docs.map((d) => ({
  id:     d.id,
  ...d.data(),
  author: authorMap.get(d.data().authorId),
}));
```

### ✅ 非正規化（Denormalization）で JOIN を避ける

Firestore では結合（JOIN）ができないため、よく一緒に参照するデータは同じドキュメントに埋め込みます。

```typescript
// 投稿ドキュメントに著者の名前・アバターを埋め込む
await addDoc(collection(db, 'posts'), {
  title:       'Hello World',
  content:     '...',
  authorId:    user.uid,
  // ↓ 著者情報を埋め込む（著者名が変わったときは更新が必要）
  authorName:  user.displayName,
  authorAvatar: user.photoURL,
  createdAt:   serverTimestamp(),
});
```

### ✅ 必要なフィールドだけ取得（`select`）

```typescript
// NG: ドキュメント全体（content が大きい場合に無駄）
const snapshot = await getDocs(collection(db, 'posts'));

// ✅ 一覧表示に必要なフィールドのみ取得
const snapshot = await getDocs(
  query(collection(db, 'posts'), orderBy('createdAt', 'desc'), limit(20))
);
// ※ Firestore の select() は SDK v9 では直接サポートなし
// → 非正規化で「posts_summary」コレクションを別途作ることも検討
```

### ✅ `limit` は必須

```typescript
// NG: 全件取得は禁止（10万件あったら壊滅的）
const snapshot = await getDocs(collection(db, 'posts'));

// ✅ 必ず件数を制限
const snapshot = await getDocs(
  query(collection(db, 'posts'), orderBy('createdAt', 'desc'), limit(20))
);
```

### ✅ カーソルページネーション

```typescript
// オフセット（skip）は使えないためカーソルベースで実装
let lastDoc: DocumentSnapshot | null = null;

async function loadNextPage() {
  const q = lastDoc
    ? query(
        collection(db, 'posts'),
        orderBy('createdAt', 'desc'),
        startAfter(lastDoc),
        limit(20)
      )
    : query(collection(db, 'posts'), orderBy('createdAt', 'desc'), limit(20));

  const snapshot = await getDocs(q);
  lastDoc = snapshot.docs[snapshot.docs.length - 1] ?? null;
  return snapshot.docs.map((d) => ({ id: d.id, ...d.data() }));
}
```

## 3. インデックス設計

### 自動インデックスと複合インデックス

| 種類 | 内容 | 設定方法 |
|------|------|---------|
| 単一フィールドインデックス | 各フィールドに自動作成 | 自動（無効化も可能） |
| 複合インデックス | 複数フィールドでの `orderBy` + `where` | `firestore.indexes.json` で定義・手動作成 |

```json
// firestore.indexes.json
{
  "indexes": [
    {
      "collectionGroup": "posts",
      "queryScope": "COLLECTION",
      "fields": [
        { "fieldPath": "authorId",  "order": "ASCENDING" },
        { "fieldPath": "createdAt", "order": "DESCENDING" }
      ]
    },
    {
      "collectionGroup": "posts",
      "queryScope": "COLLECTION",
      "fields": [
        { "fieldPath": "published",  "order": "ASCENDING" },
        { "fieldPath": "category",   "order": "ASCENDING" },
        { "fieldPath": "createdAt",  "order": "DESCENDING" }
      ]
    }
  ]
}
```

> **ヒント**: インデックスが不足しているクエリを実行すると、コンソールに必要なインデックス作成リンクが表示されます。開発中はそれをクリックするのが最速です。

### 不要なインデックスを無効化してコスト削減

```json
// firestore.indexes.json
{
  "fieldOverrides": [
    {
      "collectionGroup": "logs",
      "fieldPath": "rawData",
      "indexes": []  // rawData フィールドのインデックスを無効化（大きなテキストはインデックス不要）
    }
  ]
}
```

## 4. リアルタイム購読の最適化

### ✅ 購読スコープを絞る

```typescript
// NG: コレクション全体を購読（ドキュメントが増えるほどコスト増）
onSnapshot(collection(db, 'messages'), callback);

// ✅ 必要な範囲に絞る
onSnapshot(
  query(
    collection(db, 'messages'),
    where('roomId', '==', currentRoomId),
    orderBy('createdAt', 'desc'),
    limit(50)
  ),
  callback
);
```

### ✅ 必要な間だけ購読する

```typescript
// コンポーネントのマウント中のみ購読・アンマウント時に解除
useEffect(() => {
  const unsubscribe = onSnapshot(q, callback);
  return () => unsubscribe(); // ← 必須
}, [roomId]);
```

### ✅ `onSnapshot` vs `getDocs` の使い分け

| ケース | 推奨 |
|-------|------|
| リアルタイムでデータを更新する必要がある | `onSnapshot` |
| 画面表示時に一度だけ取得すれば十分 | `getDocs`（コスト低） |
| Server Component での初期データ取得 | Admin SDK + `get()` |

## 5. Firebase SDK のバンドルサイズ削減

Firebase SDK はモジュール単位でインポートすることでバンドルサイズを削減できます。

### Tree-Shaking を有効にする

```typescript
// NG: 名前空間インポート（バンドルサイズが大きい）
import * as firebase from 'firebase/app';
import 'firebase/firestore';
import 'firebase/auth';

// ✅ 使う関数だけインポート（Tree-Shaking が効く）
import { initializeApp } from 'firebase/app';
import { getFirestore, doc, getDoc, setDoc } from 'firebase/firestore';
import { getAuth, signInWithEmailAndPassword } from 'firebase/auth';
```

### サービスを遅延ロード（Dynamic Import）

```typescript
// 認証が必要な操作は実行時に初めてインポート
async function handleGoogleLogin() {
  // ボタンがクリックされるまで Google Auth モジュールを読み込まない
  const { getAuth, GoogleAuthProvider, signInWithPopup } = await import('firebase/auth');
  const auth     = getAuth();
  const provider = new GoogleAuthProvider();
  return signInWithPopup(auth, provider);
}
```

### バンドルサイズの確認

```bash
# Next.js のバンドル分析
npm install --save-dev @next/bundle-analyzer

# next.config.js
const withBundleAnalyzer = require('@next/bundle-analyzer')({
  enabled: process.env.ANALYZE === 'true',
});
module.exports = withBundleAnalyzer({});

# 実行
ANALYZE=true npm run build
```

### SDK ごとの概算サイズ（gzip 後）

| SDK | サイズ（gzip） | 削減のヒント |
|-----|-------------|------------|
| `firebase/app` | ~3 KB | 必須 |
| `firebase/auth` | ~35 KB | 使う Provider だけインポート |
| `firebase/firestore` | ~50 KB | 遅延ロードを検討 |
| `firebase/storage` | ~15 KB | アップロードページのみで遅延ロード |
| `firebase/functions` | ~5 KB | 小さいため通常は問題なし |

## 6. 画像・Storage の最適化

### アップロード時にリサイズ・WebP 変換

```typescript
// ブラウザ側でリサイズして Storage コストを削減
async function resizeAndUpload(file: File, maxWidth = 1200): Promise<string> {
  // Canvas でリサイズ
  const img = await createImageBitmap(file);
  const scale  = Math.min(1, maxWidth / img.width);
  const canvas = new OffscreenCanvas(img.width * scale, img.height * scale);
  const ctx    = canvas.getContext('2d')!;
  ctx.drawImage(img, 0, 0, canvas.width, canvas.height);

  // WebP に変換（品質 80%）
  const blob = await canvas.convertToBlob({ type: 'image/webp', quality: 0.8 });

  // Firebase Storage にアップロード
  const { ref, uploadBytes, getDownloadURL } = await import('firebase/storage');
  const { storage } = await import('@/lib/firebase/client');
  const storageRef = ref(storage, `images/${Date.now()}.webp`);
  await uploadBytes(storageRef, blob);
  return getDownloadURL(storageRef);
}
```

### Cloud Functions でサムネイル自動生成

```typescript
// functions/src/resizeImage.ts
import { onObjectFinalized } from 'firebase-functions/v2/storage';
import { getStorage }        from 'firebase-admin/storage';
import sharp from 'sharp';

const SIZES = [
  { suffix: '_thumb',  width: 200  },
  { suffix: '_medium', width: 800  },
];

export const resizeOnUpload = onObjectFinalized(async (event) => {
  const { bucket, name, contentType } = event.data;
  if (!contentType?.startsWith('image/')) return;  // 画像以外はスキップ
  if (SIZES.some(({ suffix }) => name!.includes(suffix))) return; // 生成済みはスキップ

  const storageBucket = getStorage().bucket(bucket);
  const [inputBuffer] = await storageBucket.file(name!).download();

  await Promise.all(
    SIZES.map(async ({ suffix, width }) => {
      const resized = await sharp(inputBuffer)
        .resize(width, undefined, { fit: 'inside', withoutEnlargement: true })
        .webp({ quality: 80 })
        .toBuffer();

      const thumbPath = name!.replace(/(\.[^/.]+)$/, `${suffix}.webp`);
      await storageBucket.file(thumbPath).save(resized, {
        contentType: 'image/webp',
        metadata: { metadata: { resizedFrom: name } },
      });
    })
  );
});
```

## 7. Cloud Functions のコールドスタート対策

### コールドスタートの原因と対策

```
コールドスタートが発生する状況:
  - 一定時間アクセスがなかった後の最初のリクエスト
  - スケールアウト時の新規インスタンス起動

対策:
  ① minInstances でウォームインスタンスを維持
  ② 初期化処理を最小化
  ③ 依存パッケージを削減
```

```typescript
// ① minInstances（コスト増に注意）
export const criticalApi = onRequest(
  { minInstances: 1, region: 'asia-northeast1' },
  handler
);

// ② グローバルスコープに重い初期化を移動（インスタンス起動時に1回だけ実行）
// NG: リクエストのたびに初期化
export const myFunction = onCall(async (req) => {
  const db = getFirestore(); // ← 毎回呼ばれる
});

// ✅ グローバルスコープで初期化（コールドスタート時の1回のみ）
const db = getFirestore();
export const myFunction = onCall(async (req) => {
  // db をそのまま使う
});
```

### メモリ・CPU のチューニング

```typescript
import { onCall } from 'firebase-functions/v2/https';

// 画像処理など重い処理: メモリを増やす
export const processImage = onCall(
  {
    memory:  '1GiB',   // デフォルト 256MB
    cpu:     2,         // デフォルト 1
    timeoutSeconds: 120,
  },
  handler
);

// 軽い API: メモリを減らしてコスト削減
export const lightApi = onCall(
  { memory: '128MiB' },
  handler
);
```

## 8. キャッシュ戦略

### Firestore のオフラインキャッシュ

```typescript
import { initializeFirestore, CACHE_SIZE_UNLIMITED, persistentLocalCache, persistentMultipleTabManager } from 'firebase/firestore';
import { getApp } from 'firebase/app';

// 永続キャッシュを有効化（デフォルトは 40MB）
const db = initializeFirestore(getApp(), {
  localCache: persistentLocalCache({
    cacheSizeBytes: CACHE_SIZE_UNLIMITED, // キャッシュサイズ無制限
    tabManager: persistentMultipleTabManager(), // 複数タブ対応
  }),
});
```

### React Query + Firestore でクライアントキャッシュ

```typescript
// lib/hooks/usePosts.ts
import { useQuery } from '@tanstack/react-query';
import { getDocs, collection, query, orderBy, limit } from 'firebase/firestore';
import { db } from '@/lib/firebase/client';

export function usePosts() {
  return useQuery({
    queryKey: ['posts'],
    queryFn: async () => {
      const snapshot = await getDocs(
        query(collection(db, 'posts'), orderBy('createdAt', 'desc'), limit(20))
      );
      return snapshot.docs.map((d) => ({ id: d.id, ...d.data() }));
    },
    staleTime: 5 * 60 * 1000,  // 5分間はキャッシュを新鮮とみなす（再取得しない）
    gcTime:    10 * 60 * 1000, // 10分後にキャッシュを破棄
  });
}
```

### Cloud Functions のレスポンスキャッシュ

```typescript
// HTTP 関数のレスポンスに Cache-Control ヘッダーを付与
export const getPublicPosts = onRequest(async (req, res) => {
  const snapshot = await adminDb
    .collection('posts')
    .where('published', '==', true)
    .orderBy('createdAt', 'desc')
    .limit(20)
    .get();

  const posts = snapshot.docs.map((d) => ({ id: d.id, ...d.data() }));

  // CDN に 60 秒キャッシュ、ユーザーブラウザに 30 秒キャッシュ
  res.set('Cache-Control', 'public, max-age=30, s-maxage=60');
  res.json({ posts });
});
```

## 9. Firebase Performance Monitoring

リアルユーザーのパフォーマンスデータを自動収集します。

### セットアップ

```bash
npm install firebase
# firebase パッケージに含まれている（追加インストール不要）
```

```typescript
// src/lib/firebase/performance.ts
import { getApp } from 'firebase/app';
import { getPerformance, trace } from 'firebase/performance';

export const perf = getPerformance(getApp());

// カスタムトレース（任意の処理の実行時間を計測）
export async function measureOperation<T>(
  traceName: string,
  operation: () => Promise<T>
): Promise<T> {
  const t = trace(perf, traceName);
  t.start();
  try {
    const result = await operation();
    return result;
  } finally {
    t.stop();
  }
}
```

```typescript
// 使用例
import { measureOperation } from '@/lib/firebase/performance';

const posts = await measureOperation('fetch_posts', () =>
  getDocs(query(collection(db, 'posts'), limit(20)))
);
```

### 自動計測される指標

- **ページロード時間**（First Contentful Paint, etc.）
- **ネットワークリクエストの応答時間**（Firestore, Functions など）
- **カスタムトレース**（独自に追加したもの）

Firebase コンソール → Performance → Dashboard で確認できます。

## まとめ・チェックリスト

### Firestore

- [ ] クエリには必ず `limit()` を付ける
- [ ] N+1 問題を解消する（`in` クエリで一括取得 or 非正規化）
- [ ] `onSnapshot` は必要な範囲に絞り、アンマウント時に解除する
- [ ] 複合インデックスを適切に設定する
- [ ] 読み取り専用データは `getDocs`（`onSnapshot` を使わない）

### SDK・バンドル

- [ ] 使う関数だけインポートして Tree-Shaking を有効にする
- [ ] 大きなモジュール（Auth, Firestore）は必要に応じて動的インポート

### Storage

- [ ] アップロード前に画像をリサイズ・WebP 変換する
- [ ] Cloud Functions でサムネイルを自動生成する

### Cloud Functions

- [ ] コールドスタートが問題なら `minInstances: 1` を設定
- [ ] 初期化処理はグローバルスコープに移動
- [ ] メモリ・CPU は処理に合わせて最小化

### キャッシュ

- [ ] Firestore の永続キャッシュを有効化
- [ ] React Query などでクライアントキャッシュを活用
- [ ] HTTP 関数には `Cache-Control` ヘッダーを付与
