## 1. ワークフロー設計

### 1.1 ワークフロー名とファイル名

**ルール**:
- ワークフロー名は短く、わかりやすく
- ファイル名はワークフロー名と同じ
- 複数の関連ワークフローには プレフィックスを使用

```yaml
# ✅ 推奨
name: CI

# .github/workflows/ci.yml

# ❌ 避けるべき
name: Run automated tests and build and deploy to servers

# .github/workflows/automated-workflow-number-1.yml
```

**複数ワークフローの場合**:

```
.github/workflows/
├── ci-lint.yml          ← Lint チェック
├── ci-test.yml          ← テスト実行
├── ci-build.yml         ← ビルド
└── cd-deploy.yml        ← デプロイ
```

### 1.2 トリガーの最小化

不要なトリガーは避け、必要なトリガーのみ指定：

```yaml
# ❌ 避けるべき: すべてのイベントで実行
on: [push, pull_request, issues, release, fork, watch]

# ✅ 推奨: 必要なトリガーのみ
on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]
```

### 1.3 ジョブの分割

関心の分離により、個別に再実行・管理可能：

```yaml
# ✅ 推奨: ジョブを分割
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - run: npm run lint
  
  test:
    runs-on: ubuntu-latest
    steps:
      - run: npm test
  
  build:
    needs: [lint, test]
    runs-on: ubuntu-latest
    steps:
      - run: npm run build

# ❌ 避けるべき: すべてを 1 つのジョブに
jobs:
  all:
    runs-on: ubuntu-latest
    steps:
      - run: npm run lint
      - run: npm test
      - run: npm run build
```

### 1.4 条件分岐の活用

```yaml
jobs:
  deploy:
    runs-on: ubuntu-latest
    # main ブランチのみ実行
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    steps:
      - run: npm run deploy

  notify-pr:
    runs-on: ubuntu-latest
    # PR のみ実行
    if: github.event_name == 'pull_request'
    steps:
      - run: echo "PR review started"

  notify-fail:
    runs-on: ubuntu-latest
    # 前のジョブが失敗した場合のみ
    if: failure()
    steps:
      - run: echo "Build failed!"
```

## 2. パフォーマンス最適化

### 2.1 キャッシュの活用

```yaml
# ✅ 推奨: 自動キャッシュ
- uses: actions/setup-node@v4
  with:
    node-version: '18'
    cache: 'npm'  # 自動で npm キャッシュを有効化

- uses: actions/setup-python@v4
  with:
    python-version: '3.11'
    cache: 'pip'  # 自動で pip キャッシュを有効化

# ✅ 手動キャッシュも有効
- uses: actions/cache@v3
  with:
    path: ~/.gradle
    key: ${{ runner.os }}-gradle-${{ hashFiles('**/gradle-wrapper.properties') }}
    restore-keys: |
      ${{ runner.os }}-gradle-
```

### 2.2 マトリックスの活用

並列テストで実行時間を短縮：

```yaml
# ❌ 避けるべき: 順序実行
jobs:
  test-16:
    steps: [test with Node 16]
  
  test-18:
    needs: test-16
    steps: [test with Node 18]
  
  test-20:
    needs: test-18
    steps: [test with Node 20]

# ✅ 推奨: 並列実行
jobs:
  test:
    strategy:
      matrix:
        node-version: [16, 18, 20]
    steps:
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
      - run: npm test
```

### 2.3 不要なステップを省く

```yaml
# ✅ 推奨: checkout は必要な場合のみ
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      # ソースコードを修正する場合のみチェックアウト
      - uses: actions/checkout@v4
        if: github.event_name == 'push'
      
      - run: npm test

# ✅ 推奨: 条件付きテスト
- name: Run expensive tests
  if: github.event_name == 'push' || contains(github.event.head_commit.message, '[test-all]')
  run: npm run test:e2e
```

### 2.4 並列実行

```yaml
jobs:
  lint:
    runs-on: ubuntu-latest
    steps: [lint tasks]
  
  test:
    runs-on: ubuntu-latest
    steps: [test tasks]
  
  build:
    runs-on: ubuntu-latest
    steps: [build tasks]

# すべてのジョブが並列で実行される（needs: がない場合）
```

## 3. セキュリティベストプラクティス

### 3.1 シークレット管理

```yaml
# ✅ 推奨: シークレットを使用
- name: Deploy
  env:
    API_KEY: ${{ secrets.API_KEY }}
  run: npm run deploy

# ❌ 避けるべき: ハードコード
- run: npm run deploy --api-key=sk-1234567890

# ❌ 避けるべき: ログに出力
- run: echo "API_KEY=${{ secrets.API_KEY }}"
  # GitHub が自動的にマスクしますが、習慣として避けるべき
```

### 3.2 アクションのバージョン管理

```yaml
# ❌ 避けるべき: main ブランチを直接参照
- uses: actions/checkout@main

# ❌ 避けるべき: バージョン未指定
- uses: actions/checkout

# ✅ 推奨: 特定のメジャーバージョンを指定
- uses: actions/checkout@v4

# ✅ 推奨: セキュリティが重要な場合は SHA 指定
- uses: actions/checkout@d632683dd7b4114ad4b0ad7147713b51f348cb18
```

### 3.3 権限の最小化

```yaml
# ✅ 推奨: 必要な権限のみ指定
jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read       # 読み取りのみ
      packages: read
      checks: write        # チェック結果の書き込み
    steps:
      - uses: actions/checkout@v4

# ❌ 避けるべき: すべての権限を与える
jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: write
      packages: write
      checks: write
      pull-requests: write
```

### 3.4 外部 PR への対策

```yaml
# ❌ 避けるべき: fork の PR でシークレットを使用
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - run: npm run deploy
        env:
          API_KEY: ${{ secrets.API_KEY }}

# ✅ 推奨: イベントタイプでフィルタ
jobs:
  deploy:
    if: github.event_name == 'push'  # push のみ
    runs-on: ubuntu-latest
    environment: production
    steps:
      - run: npm run deploy
        env:
          API_KEY: ${{ secrets.API_KEY }}

# ✅ 推奨: 別のワークフローに分割
on:
  pull_request:
    branches: [main]

jobs:
  test:  # シークレット不要
    runs-on: ubuntu-latest
    steps:
      - run: npm test

---

on:
  push:
    branches: [main]

jobs:
  deploy:  # シークレット使用
    runs-on: ubuntu-latest
    steps:
      - run: npm run deploy
        env:
          API_KEY: ${{ secrets.API_KEY }}
```

## 4. 保守性

### 4.1 ドキュメント化

```yaml
name: Build and Test

# ワークフローの目的を説明
# このワークフローは以下を実行します:
# - 依存関係のインストール
# - Lint チェック
# - テスト実行
# - ビルド

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    
    steps:
      # 各ステップに説明を追加
      - name: Checkout code
        uses: actions/checkout@v4
        with:
          fetch-depth: 0  # 全履歴を取得（差分検出用）
      
      - name: Setup Node.js with npm caching
        uses: actions/setup-node@v4
        with:
          node-version: '18'
          cache: 'npm'
```

### 4.2 DRY 原則（重複排除）

```yaml
# ❌ 避けるべき: 重複したワークフロー
# .github/workflows/test-node-16.yml
# .github/workflows/test-node-18.yml
# .github/workflows/test-node-20.yml

# ✅ 推奨: マトリックスで一元管理
jobs:
  test:
    strategy:
      matrix:
        node-version: [16, 18, 20]
    steps:
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}

# ✅ 推奨: 再利用可能ワークフロー
.github/workflows/reusable-test.yml:
  on:
    workflow_call:
      inputs:
        node-version:
          type: string
          default: '18'
  jobs:
    test:
      runs-on: ubuntu-latest
      steps:
        - uses: actions/setup-node@v4
          with:
            node-version: ${{ inputs.node-version }}
        - run: npm test

.github/workflows/ci.yml:
  jobs:
    test:
      uses: ./.github/workflows/reusable-test.yml
      with:
        node-version: '18'
```

### 4.3 バージョニング

```yaml
# ✅ 推奨: アクションのバージョンを明示
jobs:
  build:
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - uses: docker/build-push-action@v5

# バージョン更新時は定期的に確認
# $ gh workflow run ci.yml
```

### 4.4 テストの容易性

```yaml
# ✅ 推奨: ローカルでテスト可能な構造
# Dockerfile や docker-compose を提供
# ワークフローと同じコマンドをローカルで実行可能

# package.json
{
  "scripts": {
    "lint": "eslint .",
    "test": "jest",
    "build": "webpack",
    "ci": "npm run lint && npm run test && npm run build"
  }
}

# ワークフロー
- run: npm ci
- run: npm run lint
- run: npm run test
- run: npm run build
```

## 5. 監視・通知

### 5.1 失敗通知

```yaml
- name: Notify on failure
  if: failure()
  uses: slackapi/slack-github-action@v1
  with:
    webhook-url: ${{ secrets.SLACK_WEBHOOK }}
    payload: |
      {
        "text": "❌ Build failed",
        "blocks": [{
          "type": "section",
          "text": {
            "type": "mrkdwn",
            "text": "*Failed*\nBranch: ${{ github.ref }}\nCommit: ${{ github.sha }}"
          }
        }]
      }
```

### 5.2 成功時の通知

```yaml
- name: Notify on success
  if: success()
  uses: slackapi/slack-github-action@v1
  with:
    webhook-url: ${{ secrets.SLACK_WEBHOOK }}
    payload: |
      {
        "text": "✅ Build succeeded"
      }
```

### 5.3 定期的な実行確認

```yaml
# 定期的にワークフロー実行状況を確認
on:
  schedule:
    - cron: '0 9 * * 1'  # 毎週月曜 9:00

jobs:
  health-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/github-script@v7
        with:
          script: |
            const runs = await github.rest.actions.listWorkflowRuns({
              owner: context.repo.owner,
              repo: context.repo.repo,
              workflow_id: 'ci.yml',
              per_page: 10
            });
            // ワークフロー実行状況を確認
```

## 6. コスト最適化

### 6.1 実行時間の削減

```yaml
# ✅ 推奨: キャッシュで高速化
- uses: actions/cache@v3
  with:
    path: ~/.npm
    key: ${{ runner.os }}-npm-${{ hashFiles('**/package-lock.json') }}

# ✅ 推奨: 不要なテストをスキップ
- name: Run expensive tests
  if: github.ref == 'refs/heads/main'  # main でのみ実行
  run: npm run test:e2e

# ✅ 推奨: 並列実行
strategy:
  matrix:
    test-suite: [unit, integration, e2e]
```

### 6.2 不要なワークフローの削除

```yaml
# ❌ 避けるべき: 使用されていないワークフロー
.github/workflows/
├── old-deploy.yml      # 使用されていない
├── experimental.yml    # テスト用だけ
└── ci.yml              # 実際に使用

# ✅ 推奨: 定期的に確認・削除
# Settings → Actions → All workflows で確認
# 古いワークフローは削除
```

### 6.3 ランナー選択

```yaml
# ✅ 推奨: 最小限の環境で実行
jobs:
  test:
    runs-on: ubuntu-latest  # 安価で十分
    # ❌ 避けるべき: macos-latest（高コスト）
    # ❌ 避けるべき: windows-latest（高コスト）
```

## 7. よくあるアンチパターン

| アンチパターン | 問題 | 解決策 |
|----------------|------|--------|
| **シークレット値をハードコード** | セキュリティ危険 | シークレットを使用 |
| **main ブランチでアクション参照** | 予期しない変更 | バージョン指定 |
| **すべてを 1 つのジョブに** | 再実行・デバッグが困難 | ジョブを分割 |
| **キャッシュを使わない** | 実行時間が長い | キャッシュを活用 |
| **外部 PR でシークレット使用** | セキュリティ危険 | イベント型でフィルタ |
| **ドキュメント化なし** | 保守が困難 | コメント・ドキュメント追加 |

## 📌 推奨設定テンプレート

```yaml
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

permissions:
  contents: read
  checks: write

jobs:
  # Lint チェック
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '18'
          cache: 'npm'
      - run: npm ci
      - run: npm run lint

  # テスト
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version: [16, 18, 20]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'npm'
      - run: npm ci
      - run: npm test
      - uses: codecov/codecov-action@v3
        if: success()

  # ビルド
  build:
    needs: [lint, test]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '18'
          cache: 'npm'
      - run: npm ci
      - run: npm run build
      - uses: actions/upload-artifact@v3
        with:
          name: build
          path: dist/

  # デプロイ（main のみ）
  deploy:
    needs: build
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4
      - uses: actions/download-artifact@v3
        with:
          name: build
      - run: npm run deploy
        env:
          API_KEY: ${{ secrets.API_KEY }}
```
