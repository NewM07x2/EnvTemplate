## 1. GitHub Actions とは

### 1.1 GitHub Actions の定義

**GitHub Actions** は、GitHub リポジトリにおいて **自動化されたワークフロー** を実行するサービスです。

```
┌─────────────────────────────────────────┐
│    GitHub Actions の役割                │
├─────────────────────────────────────────┤
│  コード → テスト → ビルド → デプロイ    │
│  ↓        ↓      ↓       ↓              │
│  自動    自動    自動    自動            │
│  実行    実行    実行    実行            │
└─────────────────────────────────────────┘
```

### 1.2 なぜ必要か

| 課題 | GitHub Actions での解決 |
|------|------------------------|
| **手作業でテストを実行している** | 自動でテスト実行 → 失敗したら即座に通知 |
| **本番環境へのデプロイが不安** | 自動チェック後にデプロイ → ヒューマンエラー削減 |
| **チーム開発で品質がばらつく** | 誰が Push しても同じテストが実行される |
| **リリース前の確認漏れがある** | ワークフロー化して自動確認 |
| **複数の環境に対応させるのが煩雑** | 環境ごとにワークフローを設定可能 |

## 2. CI/CD の基本概念

### 2.1 CI（継続的インテグレーション）

**Continuous Integration** = コード変更を頻繁に統合し、自動的に検証するプロセス

```
開発者A          開発者B          開発者C
  │                │                │
  └────Push────────┴────Push────────┘
        ↓
   GitHub リポジトリ
        ↓
   CI が自動起動
   ├── テスト実行
   ├── 型チェック
   ├── Lint 実行
   └── コード品質分析
        ↓
   ✅ OK → マージ可能
   ❌ NG → マージ不可（修正を促す）
```

**メリット**：
- 問題を早期に発見できる
- マージ前に品質を保証できる
- チーム全体が同じ品質基準で開発できる

### 2.2 CD（継続的デプロイ）

**Continuous Deployment/Delivery** = 検証済みコードを自動的に本番環境へ反映

```
CI 完了
  ↓
  ├─ Continuous Delivery
  │   └── 本番へのデプロイ準備完了（手動で最終確認後デプロイ）
  │
  └─ Continuous Deployment
      └── 自動で本番環境へデプロイ（検証後、自動的に反映）
```

**メリット**：
- リリース頻度を高められる
- ユーザーへの価値提供が早くなる
- デプロイのリスクが低下する（小さな変更を頻繁にデプロイ）

## 3. GitHub Actions の利点

| 利点 | 説明 |
|------|------|
| **GitHub に統合** | リポジトリ内に設定できる（別サービスが不要） |
| **無料** | パブリックリポジトリは無制限、プライベートリポジトリは月 2,000 分の実行時間が無料 |
| **簡単な設定** | YAML ファイルで記述するだけ（複雑なセットアップ不要） |
| **豊富なアクション** | 数千のコミュニティアクションが利用可能 |
| **マトリックス実行** | 複数の Node.js バージョンなど、並列テストが簡単 |
| **大規模なログ保存** | 実行ログが 90 日間保存される |

## 4. 基本用語の解説

### ワークフロー（Workflow）
- **定義**: GitHub リポジトリで実行する自動化処理全体
- **ファイル位置**: `.github/workflows/*.yml`
- **例**: テスト → ビルド → デプロイ

### トリガー（Trigger）
- **定義**: ワークフローを実行するきっかけ
- **例**: 
  - `push`: コードを Push したとき
  - `pull_request`: Pull Request を作成したとき
  - `schedule`: 定時実行（cron 形式）

### ジョブ（Job）
- **定義**: ワークフロー内の実行単位
- **特徴**: 複数のジョブを並列実行できる
- **例**: テストジョブ、ビルドジョブ

### ステップ（Step）
- **定義**: ジョブ内の個別の処理
- **例**: 
  - Node.js をセットアップ
  - 依存関係をインストール
  - テストを実行

### アクション（Action）
- **定義**: ステップで再利用できる処理の集合
- **種類**:
  - 公式アクション（GitHub 提供）
  - コミュニティアクション（GitHub Marketplace）
  - カスタムアクション（自作）

### ランナー（Runner）
- **定義**: ワークフローを実行するマシン
- **種類**:
  - **GitHub ホステッド**: GitHub が提供する VM（Ubuntu, Windows, macOS）
  - **セルフホステッド**: 自分たちで管理するマシン

## 5. 料金体系

### パブリックリポジトリ
- **すべてのワークフロー実行**: **無料**（無制限）

### プライベートリポジトリ
| 実行環境 | 月額無料分 | 追加料金 |
|---------|-----------|---------|
| Ubuntu | 2,000 分 | $0.008 / 分 |
| Windows | 2,000 分（3 倍のカウント） | $0.024 / 分 |
| macOS | 2,000 分（10 倍のカウント） | $0.08 / 分 |

**例**：
- Ubuntu でテスト実行: 10 分 × 5 回 = 50 分（無料枠内）
- macOS でテスト実行: 10 分 × 2 回 = 200 分カウント（無料）

## 6. 動作環境

### GitHub ホステッド ランナーのスペック

| 項目 | Ubuntu | Windows | macOS |
|------|--------|---------|-------|
| **OS** | Ubuntu 22.04 / 20.04 | Windows Server 2022 / 2019 | macOS 13 / 12 |
| **CPU** | 2 コア | 2 コア | 3 コア |
| **メモリ** | 7 GB | 7 GB | 14 GB |
| **ディスク** | 14 GB | 14 GB | 14 GB |
| **タイムアウト** | 360 分（6 時間） | 360 分 | 360 分 |

### インストール済みソフトウェア

各ランナーに以下がプリインストールされています：

**一般的**:
- Git, Node.js, Python, Ruby, Java, Go

**ビルドツール**:
- npm, yarn, pip, Maven, Gradle

**その他**:
- Docker, Docker Compose, Terraform
