## 1. 環境変数とシークレットの違い

| 特性 | 環境変数 | シークレット |
|------|---------|------------|
| **公開性** | リポジトリに表示される | ログに表示されない（マスク） |
| **用途** | API URL、ポート番号など | パスワード、トークン、API キー |
| **セキュリティ** | 低（誰でも見える） | 高（暗号化・マスク） |
| **使用例** | `NODE_ENV=production` | `DATABASE_PASSWORD=secret123` |

## 2. 環境変数（Environment Variables）

### 2.1 ワークフローレベル

ワークフロー全体で使える環境変数：

```yaml
name: Build with environment variables

# ワークフロー全体で有効
env:
  NODE_ENV: production
  API_URL: https://api.example.com
  TIMEOUT: 30

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Show variables
        run: |
          echo "Environment: ${{ env.NODE_ENV }}"
          echo "API URL: ${{ env.API_URL }}"
          echo "Timeout: ${{ env.TIMEOUT }}"
```

### 2.2 ジョブレベル

特定のジョブ内のみで有効：

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    env:
      BUILD_TYPE: release
      VERSION: 1.0.0
    
    steps:
      - run: echo "Building $BUILD_TYPE version $VERSION"
  
  test:
    runs-on: ubuntu-latest
    env:
      PYTHONPATH: /app
    
    steps:
      - run: python test.py
```

### 2.3 ステップレベル

個別のステップのみで有効：

```yaml
steps:
  - name: Build step
    env:
      BUILD_SCRIPT: build.sh
      DEBUG: true
    run: bash $BUILD_SCRIPT
  
  - name: Test step
    env:
      TEST_DATABASE: test.db
    run: npm test
```

**環境変数のスコープ**:

```yaml
env:
  GLOBAL: "ワークフロー全体で使用可"

jobs:
  job1:
    env:
      JOB_VAR: "job1 内のみ"
    steps:
      - env:
          STEP_VAR: "このステップのみ"
        run: |
          echo $GLOBAL      # ✅ 使用可
          echo $JOB_VAR     # ✅ 使用可
          echo $STEP_VAR    # ✅ 使用可
  
  job2:
    steps:
      - run: |
          echo $GLOBAL      # ✅ 使用可
          echo $JOB_VAR     # ❌ エラー（job1 のみ）
          echo $STEP_VAR    # ❌ エラー
```

## 3. シークレット（Secrets）

機密情報（API キー、パスワード、トークン）を安全に管理します。

### 3.1 GitHub UI からの設定

**リポジトリ設定から追加**:

1. リポジトリ → 「Settings」
2. 左側メニュー → 「Secrets and variables」 → 「Actions」
3. 「New repository secret」をクリック
4. 以下を入力：
   - **Name**: `API_KEY` （大文字推奨）
   - **Value**: `sk-1234567890abcdef...`
5. 「Add secret」をクリック

**セットアップ例**:

```
Secret名                      値
─────────────────────────────────────────
DATABASE_PASSWORD             mypassword123
AWS_ACCESS_KEY_ID             AKIAIOSFODNN7EXAMPLE
AWS_SECRET_ACCESS_KEY         wJalrXUtnFEMI/K7MDENG+bPxRfiCYEXAMPLEKEY
GITHUB_TOKEN                  ghp_xxxxxxxxxxxx
SLACK_WEBHOOK                 https://hooks.slack.com/...
DOCKER_REGISTRY_PASSWORD      myregistry-password
```

### 3.2 GitHub CLI からの設定

```powershell
# シークレットを設定
gh secret set API_KEY --body "sk-1234567890abcdef"

# ファイルから読み込み
gh secret set DATABASE_PASSWORD < ./password.txt

# リスト表示
gh secret list

# シークレットを削除
gh secret delete API_KEY

# 別のリポジトリに設定（所有権が必要）
gh secret set API_KEY --repo username/repo-name --body "value"
```

### 3.3 ワークフロー内での使用

```yaml
name: Use secrets

on: push

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      # 方法 1: secrets コンテキストで使用
      - name: Deploy with API key
        run: |
          curl -H "Authorization: Bearer ${{ secrets.API_KEY }}" \
               https://api.example.com/deploy
      
      # 方法 2: 環境変数として設定
      - name: Deploy with env
        env:
          API_KEY: ${{ secrets.API_KEY }}
          DB_PASSWORD: ${{ secrets.DATABASE_PASSWORD }}
        run: ./deploy.sh
      
      # 方法 3: アクションにパス
      - uses: azure/login@v1
        with:
          creds: ${{ secrets.AZURE_CREDENTIALS }}
```

**シークレット出力時の自動マスク**:

```yaml
steps:
  - name: Show secret
    run: echo "Token: ${{ secrets.API_KEY }}"
    # 出力: Token: ***（ログではマスクされる）
```

### 3.4 シークレットの削除

GitHub UI:
1. Settings → Secrets → 削除したいシークレット → Delete

GitHub CLI:
```powershell
gh secret delete API_KEY
```

## 4. Environments（環境別シークレット）

本番環境・ステージング環境など、環境ごとにシークレットを分けて管理。

### 4.1 Environment の作成

1. リポジトリ → Settings → Environments
2. 「New environment」
3. Name: `production` （または `staging`, `development`）
4. 「Configure environment」

### 4.2 Environment レベルのシークレット

各 Environment に異なるシークレットを設定：

```
Environment: staging
├── DATABASE_URL: postgresql://staging-db.example.com
├── API_KEY: staging-api-key-xxx
└── SLACK_WEBHOOK: https://hooks.slack.com/staging

Environment: production
├── DATABASE_URL: postgresql://prod-db.example.com
├── API_KEY: prod-api-key-yyy
└── SLACK_WEBHOOK: https://hooks.slack.com/prod
```

**ワークフローで environment を指定**:

```yaml
name: Deploy

on:
  push:
    branches: [main]

jobs:
  deploy-staging:
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to staging
        env:
          DATABASE_URL: ${{ secrets.DATABASE_URL }}
          API_KEY: ${{ secrets.API_KEY }}
        run: npm run deploy:staging
  
  deploy-production:
    runs-on: ubuntu-latest
    environment: production
    needs: deploy-staging
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to production
        env:
          DATABASE_URL: ${{ secrets.DATABASE_URL }}
          API_KEY: ${{ secrets.API_KEY }}
        run: npm run deploy:prod
```

### 4.3 Environment での承認ルール

本番環境デプロイ前に、指定ユーザーの承認を必須化：

1. Environment 設定 → Deployment branches
2. 「Add deployment branch rule」
3. ブランチを指定（例: `main`）
4. 「Required reviewers」で承認者を指定

```yaml
name: Manual approval deploy

on:
  push:
    branches: [main]

jobs:
  deploy-production:
    environment:
      name: production
      url: https://example.com
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm run deploy:prod
      # ここまで来るのに承認待ちが必要
```

## 5. リポジトリ変数（Repository Variables）

非機密情報を変数として管理（GitHub Enterprise のみ）：

### 5.1 リポジトリ変数の設定

Settings → Variables → New repository variable

```
Variable name     Value
─────────────────────────────────────
NODE_VERSION      18
DEPLOY_REGION     us-east-1
DOCKER_REGISTRY   myregistry.azurecr.io
```

### 5.2 リポジトリ変数の使用

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ vars.NODE_VERSION }}
      
      - name: Build and push
        env:
          REGISTRY: ${{ vars.DOCKER_REGISTRY }}
        run: docker push $REGISTRY/my-app:latest
```

## 6. 組織レベルの共有シークレット

複数のリポジトリで同じシークレットを共有：

**設定場所**: Organization → Settings → Secrets

```yaml
# 組織レベルのシークレットを使用
- name: Deploy
  run: npm run deploy
  env:
    API_KEY: ${{ secrets.ORG_API_KEY }}  # 組織レベル
    REPO_SECRET: ${{ secrets.API_KEY }}  # リポジトリレベル
```

**優先順位**:
1. リポジトリレベルのシークレット（最優先）
2. 組織レベルのシークレット

## 7. セキュリティベストプラクティス

### 7.1 シークレットのマスキング

GitHub は自動的にログにシークレットを出力しません：

```yaml
steps:
  - name: API call
    run: |
      # ❌ シークレット値がログに出力されない（自動マスク）
      curl -H "Authorization: Bearer ${{ secrets.API_KEY }}" \
           https://api.example.com/data
    # ログに表示: Authorization: Bearer ***
```

**カスタムマスキング**:

```yaml
steps:
  - name: Custom masking
    run: |
      echo "::add-mask::my-secret-value"
      echo "Masked value: my-secret-value"  # ログに *** で表示
```

### 7.2 アクセス制御

```yaml
# ✅ 推奨: public リポジトリではシークレット不要
on:
  push:
    branches: [main]

# ❌ 避けるべき: fork された PR ではシークレット未設定
on:
  pull_request:
    branches: [main]

jobs:
  build:
    # pull_request トリガーでは、external fork の PR では
    # シークレットにアクセスできません（セキュリティ対策）
    steps:
      - run: echo ${{ secrets.API_KEY }}
        # fork の PR からこのステップは実行されない
```

### 7.3 危険な操作の回避

```yaml
# ❌ 危険: 外部 PR のコードを信頼してはいけない
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ github.event.pull_request.head.sha }}
      - run: npm test
      # fork の PR は、リポジトリのシークレットにアクセスできません

# ✅ 推奨: 外部 PR は別のワークフロー・環境で実行
on:
  pull_request:
    branches: [main]

jobs:
  test:
    # シークレットを使わない
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm test  # パブリックなテストのみ

  upload-coverage:
    needs: test
    if: github.event_name == 'push'  # push のみ
    runs-on: ubuntu-latest
    steps:
      # シークレットを使う処理
      - uses: codecov/codecov-action@v3
        with:
          token: ${{ secrets.CODECOV_TOKEN }}
```

## 8. トラブルシューティング

### 8.1 シークレットが認識されない

**原因**: シークレット名のスペルミス

```yaml
# ❌ シークレット名が存在しない
run: echo ${{ secrets.API_KAY }}  # "API_KAY" は存在しない
# 結果: 空文字列

# ✅ 正しいシークレット名
run: echo ${{ secrets.API_KEY }}
```

**確認方法**:

```powershell
# リポジトリのシークレット一覧を確認
gh secret list
```

### 8.2 シークレットが無視される

**原因**: Pull Request トリガーで外部 fork からの場合

```yaml
on:
  pull_request:  # fork の PR ではシークレット不可

jobs:
  deploy:
    steps:
      - run: echo ${{ secrets.API_KEY }}
        # fork の PR: 空文字列
        # リポジトリの push: 正常に動作
```

**解決策**:

```yaml
# fork の PR と push を分ける
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm test

  deploy:
    # push のみで実行
    if: github.event_name == 'push'
    needs: test
    runs-on: ubuntu-latest
    steps:
      - run: npm run deploy
        env:
          API_KEY: ${{ secrets.API_KEY }}
```

### 8.3 Pull Request で外部ユーザーがシークレットにアクセスできない

これは **意図的なセキュリティ対策** です。fork された PR では、悪意あるコードがシークレットを盗むのを防ぎます。
