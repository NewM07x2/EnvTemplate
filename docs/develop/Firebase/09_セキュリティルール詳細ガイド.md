## 1. セキュリティルールとは

Firebase のセキュリティルールは、**クライアントから直接 Firestore や Cloud Storage にアクセスする際のアクセス制御**を定義するものです。

```
┌─────────────┐   読み書きリクエスト    ┌──────────────────────┐
│   クライアント  │ ──────────────────→  │ セキュリティルール     │
│  （ブラウザ）  │                       │ ・認証チェック         │
└─────────────┘                       │ ・データ検証           │
                                       │ ・権限チェック         │
                                       └──────────┬───────────┘
                                           ✅ 許可  │  ❌ 拒否
                                                   ▼
                                       ┌──────────────────────┐
                                       │  Firestore / Storage  │
                                       └──────────────────────┘
```

> **重要**: Admin SDK を使ったサーバーサイドアクセス（Route Handler・Cloud Functions）はセキュリティルールをバイパスします。クライアント SDK 経由のアクセスにのみ適用されます。

## 2. Firestore セキュリティルールの基本構文

```
// firestore.rules
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // コレクションへのマッチ
    match /posts/{postId} {
      allow read:  if <条件>;
      allow write: if <条件>;

      // write は create / update / delete に分割できる
      allow create: if <条件>;
      allow update: if <条件>;
      allow delete: if <条件>;
    }
  }
}
```

### 主要なオペレーション

| オペレーション | 説明 | 対応する操作 |
|-------------|------|------------|
| `read` | 読み取り全般 | `get` + `list` |
| `get` | 単一ドキュメントの取得 | `getDoc` |
| `list` | コレクションの一覧取得 | `getDocs`, `onSnapshot` |
| `write` | 書き込み全般 | `create` + `update` + `delete` |
| `create` | ドキュメントの新規作成 | `addDoc`, `setDoc`（新規） |
| `update` | 既存ドキュメントの更新 | `updateDoc`, `setDoc({merge:true})` |
| `delete` | ドキュメントの削除 | `deleteDoc` |

## 3. 認証情報を使ったアクセス制御

### `request.auth` オブジェクト

| プロパティ | 型 | 説明 |
|-----------|---|------|
| `request.auth` | object \| null | 未認証は `null` |
| `request.auth.uid` | string | ユーザーの一意 ID |
| `request.auth.email` | string | メールアドレス |
| `request.auth.email_verified` | boolean | メール確認済みか |
| `request.auth.token` | map | カスタムクレームを含む JWT トークン |

### 基本的なパターン

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // ① 全員読み取り可能・ログイン済みのみ書き込み可能
    match /posts/{postId} {
      allow read:  if true;
      allow write: if request.auth != null;
    }

    // ② 自分のデータのみ読み書き可能
    match /users/{userId} {
      allow read, write: if request.auth != null
                         && request.auth.uid == userId;
    }

    // ③ 作成者のみ更新・削除可能（他人は読み取りのみ）
    match /articles/{articleId} {
      allow read: if true;
      allow create: if request.auth != null;
      allow update, delete: if request.auth != null
                            && request.auth.uid == resource.data.authorId;
    }

    // ④ メール確認済みのユーザーのみ書き込み可能
    match /comments/{commentId} {
      allow read: if true;
      allow create: if request.auth != null
                   && request.auth.token.email_verified == true;
    }
  }
}
```

### `resource` と `request.resource` の違い

| 変数 | 説明 | 使用タイミング |
|------|------|-------------|
| `resource.data` | 現在 DB にあるデータ | `read`, `update`, `delete` で使用 |
| `request.resource.data` | クライアントが書き込もうとしているデータ | `create`, `update` で使用 |

```
// 更新前後のデータを比較する例
match /posts/{postId} {
  allow update: if request.auth.uid == resource.data.authorId  // 既存の authorId で判定
               && request.resource.data.authorId == resource.data.authorId; // authorId 変更を禁止
}
```

## 4. データ検証（フィールドの型・値チェック）

### 型チェック

```
match /posts/{postId} {
  allow create: if request.auth != null
               && request.resource.data.title is string          // string 型
               && request.resource.data.title.size() <= 100      // 100文字以内
               && request.resource.data.content is string
               && request.resource.data.content.size() <= 5000
               && request.resource.data.published is bool        // boolean 型
               && request.resource.data.viewCount is int         // 整数型
               && request.resource.data.tags is list;            // 配列型
}
```

### 型チェック関数一覧

| 関数 | 説明 |
|------|------|
| `value is string` | 文字列か |
| `value is int` | 整数か |
| `value is float` | 浮動小数点か |
| `value is number` | 数値（int または float）か |
| `value is bool` | 真偽値か |
| `value is timestamp` | タイムスタンプか |
| `value is list` | 配列か |
| `value is map` | マップ（オブジェクト）か |
| `value is null` | null か |
| `value is path` | ドキュメントパスか |

### 必須フィールドの検証

```
match /posts/{postId} {
  allow create: if request.auth != null
               // 必須フィールドがすべて含まれているか確認
               && request.resource.data.keys().hasAll(['title', 'content', 'authorId'])
               // 余分なフィールドが含まれていないか確認
               && request.resource.data.keys().hasOnly(['title', 'content', 'authorId', 'tags', 'createdAt']);
}
```

### 更新時のフィールド変更禁止

```
match /posts/{postId} {
  allow update: if request.auth.uid == resource.data.authorId
               // createdAt と authorId の変更を禁止
               && !('createdAt' in request.resource.data.diff(resource.data).affectedKeys())
               && !('authorId' in request.resource.data.diff(resource.data).affectedKeys());
}
```

## 5. 関数を使った再利用

同じ条件を複数の場所で使う場合、関数に切り出して再利用できます。

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // ── ヘルパー関数 ─────────────────────────────────────
    // ログイン済みか
    function isSignedIn() {
      return request.auth != null;
    }

    // 自分自身のドキュメントか
    function isOwner(userId) {
      return isSignedIn() && request.auth.uid == userId;
    }

    // ドキュメントの作成者か
    function isAuthor() {
      return isSignedIn() && request.auth.uid == resource.data.authorId;
    }

    // 管理者か（カスタムクレームを利用）
    function isAdmin() {
      return isSignedIn() && request.auth.token.role == 'admin';
    }

    // 有効な投稿データか（バリデーション）
    function isValidPost() {
      let data = request.resource.data;
      return data.title is string
          && data.title.size() >= 1
          && data.title.size() <= 100
          && data.content is string
          && data.content.size() <= 5000;
    }

    // ── ルール ────────────────────────────────────────────
    match /users/{userId} {
      allow read:   if isSignedIn();
      allow write:  if isOwner(userId) || isAdmin();
    }

    match /posts/{postId} {
      allow read:   if true;
      allow create: if isSignedIn() && isValidPost();
      allow update: if (isAuthor() || isAdmin()) && isValidPost();
      allow delete: if isAuthor() || isAdmin();
    }
  }
}
```

## 6. ロールベースアクセス制御（RBAC）

### パターン①: カスタムクレームを使う（推奨）

管理者コンソールや Cloud Functions でカスタムクレームを設定します。

```typescript
// Cloud Functions でロールを付与
import { getAuth } from 'firebase-admin/auth';

export const setAdminRole = onCall(async (request) => {
  // 既存の管理者のみ実行可能
  if (request.auth?.token.role !== 'admin') {
    throw new HttpsError('permission-denied', '権限がありません');
  }

  const { uid } = request.data;
  await getAuth().setCustomUserClaims(uid, { role: 'admin' });
  return { message: '管理者権限を付与しました' };
});
```

```
// セキュリティルールでカスタムクレームを確認
match /admin-content/{docId} {
  allow read, write: if request.auth.token.role == 'admin';
}
```

### パターン②: Firestore の `users` コレクションを参照

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // users コレクションからロールを取得
    function getUserRole() {
      return get(/databases/$(database)/documents/users/$(request.auth.uid)).data.role;
    }

    function isAdmin() {
      return request.auth != null && getUserRole() == 'admin';
    }

    function isModerator() {
      return request.auth != null && getUserRole() in ['admin', 'moderator'];
    }

    match /posts/{postId} {
      allow read:   if true;
      allow create: if request.auth != null;
      allow update: if request.auth.uid == resource.data.authorId || isModerator();
      allow delete: if request.auth.uid == resource.data.authorId || isAdmin();
    }
  }
}
```

> **注意**: `get()` 関数で他のドキュメントを参照するとその分の読み取りコストが発生します。頻繁にアクセスされるルールではカスタムクレームの方が効率的です。

## 7. サブコレクションのルール

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    match /posts/{postId} {
      allow read: if true;
      allow write: if request.auth != null;

      // サブコレクション（comments）のルール
      match /comments/{commentId} {
        allow read: if true;
        allow create: if request.auth != null
                     && request.resource.data.text is string
                     && request.resource.data.text.size() <= 500;
        // コメント作成者のみ削除可能
        allow delete: if request.auth.uid == resource.data.authorId;
        // 更新は禁止（不変）
        allow update: if false;
      }

      // さらに深いサブコレクションにも一括マッチ
      match /{subCollection=**} {
        allow read: if request.auth != null;
        allow write: if false; // 明示的に定義されたもの以外は禁止
      }
    }
  }
}
```

### ワイルドカードまとめ

| パターン | 説明 | 例 |
|---------|------|---|
| `{id}` | 1 階層のワイルドカード | `/posts/{postId}` |
| `{path=**}` | 0 階層以上のワイルドカード（再帰） | `/posts/{postId}/{path=**}` |

## 8. Cloud Storage セキュリティルール

```
// storage.rules
rules_version = '2';
service firebase.storage {
  match /b/{bucket}/o {

    // ── ヘルパー関数 ──────────────────────────────────────
    function isSignedIn() {
      return request.auth != null;
    }

    function isOwner(userId) {
      return request.auth.uid == userId;
    }

    // ── ユーザーのプロフィール画像 ──────────────────────────
    match /users/{userId}/profile/{fileName} {
      // 全員が閲覧可能
      allow read: if true;
      // 本人のみアップロード可能
      allow write: if isOwner(userId)
                  // 5MB 以内
                  && request.resource.size <= 5 * 1024 * 1024
                  // 画像ファイルのみ
                  && request.resource.contentType.matches('image/.*');
    }

    // ── 投稿の添付ファイル ───────────────────────────────
    match /posts/{postId}/{fileName} {
      allow read: if true;
      allow create: if isSignedIn()
                   // 20MB 以内
                   && request.resource.size <= 20 * 1024 * 1024
                   // 画像・PDF のみ
                   && request.resource.contentType.matches('image/.*|application/pdf');
      // 作成者のみ削除可能（メタデータの authorId を確認）
      allow delete: if isOwner(resource.metadata.authorId);
    }

    // ── プライベートファイル ──────────────────────────────
    match /private/{userId}/{allPaths=**} {
      // 本人のみアクセス可能
      allow read, write: if isOwner(userId);
    }
  }
}
```

### Storage ルールの主要変数

| 変数 | 説明 |
|------|------|
| `request.resource.size` | アップロードするファイルのバイト数 |
| `request.resource.contentType` | アップロードするファイルの MIME タイプ |
| `request.resource.metadata` | アップロード時に付与したメタデータ |
| `resource.size` | 既存ファイルのバイト数 |
| `resource.contentType` | 既存ファイルの MIME タイプ |
| `resource.metadata` | 既存ファイルのメタデータ |

## 9. ルールのテスト方法

### Firebase Emulator でのテスト

```bash
# エミュレーター起動
firebase emulators:start --only firestore,storage

# ルールの変更を反映（再起動不要）
firebase emulators:start --only firestore --rules firestore.rules
```

### ルールユニットテスト（`@firebase/rules-unit-testing`）

```bash
npm install --save-dev @firebase/rules-unit-testing
```

```typescript
// tests/firestore.rules.test.ts
import {
  assertFails,
  assertSucceeds,
  initializeTestEnvironment,
  RulesTestEnvironment,
} from '@firebase/rules-unit-testing';
import { doc, getDoc, setDoc, deleteDoc } from 'firebase/firestore';
import { readFileSync } from 'fs';

let testEnv: RulesTestEnvironment;

beforeAll(async () => {
  testEnv = await initializeTestEnvironment({
    projectId: 'test-project',
    firestore: {
      rules: readFileSync('firestore.rules', 'utf8'),
      host: '127.0.0.1',
      port: 8080,
    },
  });
});

afterAll(async () => {
  await testEnv.cleanup();
});

afterEach(async () => {
  await testEnv.clearFirestore();
});

describe('posts コレクション', () => {
  test('誰でも読み取れる', async () => {
    const unauthContext = testEnv.unauthenticatedContext();
    const db = unauthContext.firestore();
    await assertSucceeds(getDoc(doc(db, 'posts', 'post1')));
  });

  test('未認証ユーザーは書き込めない', async () => {
    const unauthContext = testEnv.unauthenticatedContext();
    const db = unauthContext.firestore();
    await assertFails(setDoc(doc(db, 'posts', 'post1'), { title: 'テスト' }));
  });

  test('認証済みユーザーは作成できる', async () => {
    const authContext = testEnv.authenticatedContext('user1');
    const db = authContext.firestore();
    await assertSucceeds(
      setDoc(doc(db, 'posts', 'post1'), {
        title: 'テスト投稿',
        content: '内容',
        authorId: 'user1',
      })
    );
  });

  test('他のユーザーの投稿は削除できない', async () => {
    // まず user1 でデータを作成
    await testEnv.withSecurityRulesDisabled(async (context) => {
      await setDoc(doc(context.firestore(), 'posts', 'post1'), {
        title: 'テスト',
        authorId: 'user1',
      });
    });

    // user2 で削除を試みる → 失敗するべき
    const user2Context = testEnv.authenticatedContext('user2');
    await assertFails(deleteDoc(doc(user2Context.firestore(), 'posts', 'post1')));
  });

  test('自分の投稿は削除できる', async () => {
    await testEnv.withSecurityRulesDisabled(async (context) => {
      await setDoc(doc(context.firestore(), 'posts', 'post1'), {
        title: 'テスト',
        authorId: 'user1',
      });
    });

    const user1Context = testEnv.authenticatedContext('user1');
    await assertSucceeds(deleteDoc(doc(user1Context.firestore(), 'posts', 'post1')));
  });
});
```

```bash
# テスト実行
npx jest tests/firestore.rules.test.ts
```

## 10. よくある落とし穴と対策

### ❌ デフォルトルール（全拒否）のまま公開

```
// NG: 本番ルールを書かずにデプロイすると全アクセスが拒否される
match /{document=**} {
  allow read, write: if false;
}
```

---

### ❌ 開発用の「全許可」ルールを本番に適用

```
// 絶対NG: データが完全に公開・改ざん可能になる
match /{document=**} {
  allow read, write: if true;
}
```

---

### ❌ `allow write` で `create/update/delete` を一括許可

```
// NG: 削除されるべきでないデータも削除できてしまう
match /users/{userId} {
  allow write: if request.auth.uid == userId;
}
```

```
// ✅ 操作ごとに適切な条件を設定
match /users/{userId} {
  allow create: if request.auth.uid == userId;
  allow update: if request.auth.uid == userId;
  allow delete: if false; // 論理削除のみ許可・物理削除は禁止
}
```

---

### ❌ `list` 操作を `get` と別に制御し忘れる

```
// NG: 個別取得は制限しているが一覧取得は可能
match /private-data/{docId} {
  allow get: if request.auth.uid == resource.data.ownerId;
  // list が未定義 → デフォルトは拒否だが明示すべき
}
```

```
// ✅ 明示的に list を制御
match /private-data/{docId} {
  allow get:  if request.auth.uid == resource.data.ownerId;
  allow list: if false; // 一覧取得は禁止
}
```

## まとめ

| カテゴリ | ポイント |
|---------|---------|
| 基本原則 | デフォルトは全拒否・必要なものだけ許可 |
| 認証チェック | `request.auth != null` を必ず確認 |
| 所有者チェック | `request.auth.uid == resource.data.authorId` パターン |
| データ検証 | `is string` / `size()` / `hasAll()` で型・値を検証 |
| 再利用 | 共通ロジックは関数化（`function isAdmin() {...}`） |
| ロール管理 | カスタムクレームが高速・Firestore 参照は `get()` コスト注意 |
| テスト | `@firebase/rules-unit-testing` でユニットテストを書く |
