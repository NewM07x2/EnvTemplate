## 1. スキーマ・マイグレーションテスト

### 1.1 スキーマ定義

| # | テスト観点 | 詳細 | 優先度 |
|---|-----------|------|--------|
| 1 | テーブル/コレクションの存在 | マイグレーション実行後に全テーブルが作成されていること | 🔴 最高 |
| 2 | カラム定義の正確性 | カラム名・データ型・デフォルト値・NULL 制約が仕様通りであること | 🔴 最高 |
| 3 | インデックスの存在 | 必要なインデックス（単一・複合・ユニーク・部分）が作成されていること | 🔴 最高 |
| 4 | 外部キー制約 | テーブル間のリレーション（外部キー）が正しく設定されていること | 🔴 最高 |
| 5 | CHECK 制約 | ビジネスルールに基づく CHECK 制約が設定されていること | 🟠 高い |
| 6 | ENUM / 列挙型 | ステータス等の ENUM 値が仕様通りに定義されていること | 🟠 高い |
| 7 | パーティション | 大規模テーブルのパーティション設定が正しいこと | 🟡 中 |
| 8 | トリガー / ファンクション | DB トリガーやストアドファンクションが正しく動作すること | 🟡 中 |

### 1.2 マイグレーション

| # | テスト観点 | 詳細 | 優先度 |
|---|-----------|------|--------|
| 1 | Up マイグレーション | 全マイグレーションを順次実行してスキーマが最新になること | 🔴 最高 |
| 2 | Down マイグレーション（ロールバック） | ロールバックでスキーマが一つ前の状態に戻ること | 🔴 最高 |
| 3 | 空の DB からの完全マイグレーション | ゼロベースから全マイグレーションを適用して正しいスキーマになること | 🔴 最高 |
| 4 | マイグレーションの冪等性 | 同じマイグレーションを2回実行してもエラーにならないこと | 🟠 高い |
| 5 | データを含むマイグレーション | 既存データがマイグレーション後も正しく保持されること | 🔴 最高 |
| 6 | 破壊的変更の検出 | カラム削除・型変更がデータ損失を引き起こさないこと | 🔴 最高 |
| 7 | マイグレーションの実行順序 | 依存関係のあるマイグレーションが正しい順序で実行されること | 🟠 高い |
| 8 | 本番データ量でのマイグレーション速度 | 大量データがある状態でマイグレーションが許容時間内に完了すること | 🟠 高い |

## 2. CRUD 操作テスト

### 2.1 Create（挿入）

| # | テスト観点 | 詳細 | 優先度 |
|---|-----------|------|--------|
| 1 | 正常な挿入 | 全フィールドを指定してレコードが作成されること | 🔴 最高 |
| 2 | 必須フィールドのみで挿入 | オプショナルフィールドを省略して挿入できること | 🟠 高い |
| 3 | デフォルト値の適用 | 未指定のフィールドにデフォルト値が設定されること | 🟠 高い |
| 4 | AUTO INCREMENT / UUID | 主キーが自動生成されること | 🔴 最高 |
| 5 | created_at / updated_at | タイムスタンプが自動設定されること | 🟠 高い |
| 6 | バルクインサート | 大量レコードの一括挿入が正しく動作すること | 🟠 高い |
| 7 | UPSERT（INSERT ON CONFLICT） | 競合時に更新またはスキップが正しく動作すること | 🟠 高い |
| 8 | RETURNING 句 | 挿入後に生成された値が正しく返却されること | 🟡 中 |

### 2.2 Read（読み取り）

| # | テスト観点 | 詳細 | 優先度 |
|---|-----------|------|--------|
| 1 | 主キーによる取得 | 単一レコードの取得が正しいこと | 🔴 最高 |
| 2 | 全件取得 | 全レコードのリストが正しいこと | 🔴 最高 |
| 3 | 条件付き取得（WHERE） | フィルタ条件に合致するレコードのみ取得されること | 🔴 最高 |
| 4 | ソート（ORDER BY） | 昇順/降順の並び替えが正しいこと（NULL の扱い含む） | 🟠 高い |
| 5 | ページネーション（LIMIT / OFFSET） | ページ分割が正しく動作すること | 🟠 高い |
| 6 | カーソルベースページネーション | カーソル(ID/タイムスタンプ)を使った分割が正しいこと | 🟠 高い |
| 7 | JOIN（内部結合/外部結合） | テーブル結合の結果が正しいこと | 🔴 最高 |
| 8 | サブクエリ | サブクエリの結果が正しいこと | 🟠 高い |
| 9 | 集計（GROUP BY / HAVING） | COUNT / SUM / AVG / MAX / MIN の結果が正しいこと | 🟠 高い |
| 10 | 全文検索（LIKE / ILIKE / FTS） | テキスト検索の結果が正しいこと | 🟡 中 |
| 11 | NULL の扱い | IS NULL / IS NOT NULL の条件が正しく評価されること | 🟠 高い |
| 12 | 存在しないレコード | 該当レコードがない場合に空結果/NULL が返ること | 🔴 最高 |

### 2.3 Update（更新）

| # | テスト観点 | 詳細 | 優先度 |
|---|-----------|------|--------|
| 1 | 単一レコードの更新 | 指定フィールドのみが更新されること | 🔴 最高 |
| 2 | 部分更新（PATCH 的） | 未指定のフィールドが変更されないこと | 🔴 最高 |
| 3 | 条件付き更新 | WHERE 句に合致するレコードのみ更新されること | 🔴 最高 |
| 4 | updated_at の更新 | 更新時にタイムスタンプが自動更新されること | 🟠 高い |
| 5 | バルクアップデート | 複数レコードの一括更新が正しく動作すること | 🟠 高い |
| 6 | 楽観的ロック | version / updated_at を使ったコンフリクト検出が動作すること | 🟠 高い |
| 7 | 存在しないレコードの更新 | 該当レコードがない場合の挙動が正しいこと（0行更新 / エラー） | 🟠 高い |

### 2.4 Delete（削除）

| # | テスト観点 | 詳細 | 優先度 |
|---|-----------|------|--------|
| 1 | 単一レコードの削除 | 指定レコードが削除されること | 🔴 最高 |
| 2 | 条件付き削除 | WHERE 句に合致するレコードのみ削除されること | 🔴 最高 |
| 3 | カスケード削除 | 親レコード削除時に子レコードが連鎖削除されること | 🟠 高い |
| 4 | 制約による削除防止 | 子レコードが存在する場合に削除が拒否されること（RESTRICT） | 🟠 高い |
| 5 | ソフトデリート（論理削除） | deleted_at に日時が設定され、物理削除されないこと | 🔴 最高 |
| 6 | ソフトデリートのフィルタ | 論理削除されたレコードが通常のクエリで取得されないこと | 🔴 最高 |
| 7 | ソフトデリートの復元 | 論理削除されたレコードを復元できること | 🟡 中 |
| 8 | 物理削除（完全削除） | GDPR 等に基づく完全削除が正しく動作すること | 🟠 高い |

## 3. データ整合性・制約テスト

### 3.1 主キー・ユニーク制約

| # | テスト観点 | 詳細 | 優先度 |
|---|-----------|------|--------|
| 1 | 主キーの一意性 | 重複する主キーでの挿入が拒否されること | 🔴 最高 |
| 2 | ユニーク制約 | ユニーク制約のあるカラムに重複値が挿入できないこと | 🔴 最高 |
| 3 | 複合ユニーク制約 | 複合ユニーク制約が正しく機能すること | 🟠 高い |
| 4 | ソフトデリートとユニーク | 論理削除レコードがユニーク制約に影響しないこと（部分インデックス） | 🟠 高い |

### 3.2 外部キー制約

| # | テスト観点 | 詳細 | 優先度 |
|---|-----------|------|--------|
| 1 | 参照整合性 | 存在しない外部キーでの挿入が拒否されること | 🔴 最高 |
| 2 | CASCADE の動作 | 親レコード削除時の子レコードの処理が正しいこと | 🟠 高い |
| 3 | SET NULL の動作 | 親レコード削除時に外部キーが NULL になること | 🟠 高い |
| 4 | RESTRICT の動作 | 子レコードが存在する親の削除が防止されること | 🟠 高い |
| 5 | 自己参照 | ツリー構造などの自己参照外部キーが正しく動作すること | 🟡 中 |

### 3.3 その他の制約

| # | テスト観点 | 詳細 | 優先度 |
|---|-----------|------|--------|
| 1 | NOT NULL 制約 | NULL 非許可カラムへの NULL 挿入が拒否されること | 🔴 最高 |
| 2 | CHECK 制約 | 条件を満たさない値の挿入が拒否されること（例: 価格 >= 0） | 🟠 高い |
| 3 | デフォルト値 | 値を指定しない場合にデフォルト値が設定されること | 🟠 高い |
| 4 | データ型の範囲 | INTEGER / VARCHAR 等の型に収まらないデータが拒否されること | 🟠 高い |
| 5 | ENUM 制約 | 定義された値以外が挿入できないこと | 🟠 高い |

## 5. トランザクション・同時実行テスト

### 5.1 トランザクション

| # | テスト観点 | 詳細 | 優先度 |
|---|-----------|------|--------|
| 1 | ACID 特性 — 原子性（Atomicity） | トランザクション内の全操作が成功するか全て取り消されること | 🔴 最高 |
| 2 | ACID 特性 — 一貫性（Consistency） | トランザクション前後でデータの整合性が保たれること | 🔴 最高 |
| 3 | ACID 特性 — 独立性（Isolation） | 並行トランザクションが互いに干渉しないこと | 🔴 最高 |
| 4 | ACID 特性 — 永続性（Durability） | コミット後のデータがシステム障害後も失われないこと | 🔴 最高 |
| 5 | ロールバック | 例外発生時にトランザクションが自動ロールバックされること | 🔴 最高 |
| 6 | ネストされたトランザクション | SAVEPOINT を使ったネストトランザクションが正しく動作すること | 🟡 中 |
| 7 | トランザクションのタイムアウト | 長時間のトランザクションがタイムアウトすること | 🟠 高い |

### 5.2 同時実行制御

| # | テスト観点 | 詳細 | 優先度 |
|---|-----------|------|--------|
| 1 | 楽観的ロック | version カラムによる衝突検出が動作すること | 🔴 最高 |
| 2 | 悲観的ロック（SELECT FOR UPDATE） | ロック取得中に他トランザクションが待機すること | 🟠 高い |
| 3 | デッドロックの回避 | デッドロックが発生した場合にDBがどちらかをアボートすること | 🟠 高い |
| 4 | レースコンディション | 同時更新で在庫がマイナスにならないこと（例: 在庫管理） | 🔴 最高 |
| 5 | Phantom Read の防止 | SERIALIZABLE 分離レベルでファントムリードが発生しないこと | 🟡 中 |
| 6 | Lost Update の防止 | 同時更新で一方の更新が失われないこと | 🔴 最高 |
| 7 | コネクションプール | 高負荷時にコネクション枯渇が発生しないこと | 🟠 高い |

## 📌 DB テストの推奨ツール・手法

| 用途 | ツール / 手法 |
|------|-------------|
| テスト用 DB | Testcontainers（Docker）/ SQLite（軽量テスト） |
| マイグレーション | Prisma Migrate / TypeORM Migration / Alembic / Flyway |
| テストデータ | Factory（factory_boy / Fishery）/ Seed スクリプト |
| クエリ分析 | EXPLAIN ANALYZE / pg_stat_statements / slow_query_log |
| 負荷テスト | pgbench / sysbench / mysqlslap |
| バックアップ検証 | pg_dump + pg_restore / mysqldump |
| モニタリング | pg_stat_activity / SHOW PROCESSLIST / Redis MONITOR |

### テスト環境のベストプラクティス

```
1. テストごとにトランザクションでラップし、テスト後にロールバック
2. テスト間の依存を排除（各テストが独立して実行可能であること）
3. テストデータは Factory パターンで動的に生成
4. 本番に近いデータ量でのパフォーマンステストを定期実行
5. CI/CD で Testcontainers を使い、実際の DB エンジンでテスト
```
