## 1. 監視・モニタリング

### 1.1 Vercel Analytics

**ダッシュボード**: Project > Analytics

リアルユーザーメトリクス (RUM)：

| メトリクス | 説明 | 目標 |
|---|---|---|
| **LCP** | Largest Contentful Paint | < 2.5s |
| **FID** | First Input Delay | < 100ms |
| **CLS** | Cumulative Layout Shift | < 0.1 |
| **TTFB** | Time to First Byte | < 600ms |

```
Project > Analytics > Web Vitals
```

で過去 7 日・30 日のデータを確認。

### 1.2 Vercel Observability

**Advanced Monitoring** (Pro/Enterprise):

```
Project > Monitoring > Create Alert
```

アラート条件を設定：

```json
{
  "metric": "response_time",
  "threshold": 1000,  // ms
  "condition": ">"
}
```

### 1.3 外部モニタリング統合

#### Datadog

```bash
npm install @datadog/browser-analytics
```

```javascript
// pages/_app.jsx
import { init, context } from '@datadog/browser-analytics';

init({
  applicationId: 'your-app-id',
  clientToken: 'your-token',
  site: 'datadoghq.com',
  service: 'my-vercel-app',
  env: process.env.NODE_ENV,
});
```

#### New Relic

```bash
npm install @newrelic/browser-agent
```

```html
<!-- pages/_document.jsx -->
<script src="https://js-agent.newrelic.com/nr-spa.min.js"></script>
<script>
  window.NREUM = {
    // New Relic config
  }
</script>
```

### 1.4 ヘルスチェック

```javascript
// pages/api/health.js
export default function handler(req, res) {
  res.status(200).json({
    status: 'ok',
    timestamp: new Date(),
    version: process.env.APP_VERSION,
  });
}
```

定期的に HTTP リクエストで監視：

```bash
# Uptime monitoring (例: UptimeRobot)
https://my-app.vercel.app/api/health
```

## 2. アップデート・バージョン管理

### 2.1 Node.js のアップデート

**現在のバージョン確認:**

```bash
vercel project inspect
# nodeVersion: 18.x

# または
node --version  # ローカル
```

**新バージョンへアップデート:**

```javascript
// vercel.json
{
  "buildCommand": "npm run build",
  "builds": [
    {
      "src": "package.json",
      "use": "@vercel/node@3",
      "config": {
        "nodeVersion": "20.x"  // 18.x → 20.x
      }
    }
  ]
}
```

```bash
# ローカルも同期
nvm install 20
nvm use 20
echo "20.0.0" > .nvmrc
```

### 2.2 Next.js メジャー バージョンアップ

```bash
# 現在のバージョン確認
npm list next

# 最新版にアップグレード
npm install next@latest

# 依存関係も更新
npm install react@latest react-dom@latest

# 破壊的変更の確認
npx next@latest codemod@latest
```

**テスト手順:**

```bash
# 1. ローカルでビルド・動作確認
npm run build
npm run dev

# 2. テストスイート実行
npm test

# 3. Lighthouse スコア確認
npx lighthouse http://localhost:3000

# 4. ステージング環境で検証
git push origin develop  # Preview 環境自動デプロイ

# 5. 本番環境に反映
git merge develop main
```

### 2.3 依存関係の定期更新

```bash
# セキュリティアップデート確認
npm audit

# 脆弱性を修正
npm audit fix

# マイナーバージョン更新
npm update

# package.json 表示
npm list --depth=0

# 古いパッケージ確認
npm outdated
```

**自動更新設定:**

```
Repository > Settings > Code security and analysis > Dependabot
```

- Dependabot alerts 有効化
- Dependabot security updates 有効化
- Dependabot version updates 有効化（自動 PR 生成）

## 3. セキュリティ

### 3.1 環境変数・シークレット管理

**ベストプラクティス:**

```bash
# ✅ 正しい方法
- シークレットは Vercel の Environment Variables に保存
- .env.local は git 管理から除外
- NEXT_PUBLIC_ プレフィックスを クライアント側のみに使用

# ❌ 間違った方法
- シークレットを コード内に記載
- .env ファイルを git コミット
- 環境変数をクライアント側でログ出力
```

```bash
# .gitignore に追加
.env.local
.env.*.local
.vercel/
node_modules/
```

### 3.2 SSL/TLS 証明書

Vercel は **自動的に Let's Encrypt 証明書** を発行・更新。

**確認方法:**

```
Project > Settings > Domains
```

各ドメインに「✓ Valid SSL Certificate」表示。

カスタム証明書の場合：

```
Project > Settings > Domains > [Domain] > Manage > SSL/TLS > Upload Custom SSL Certificate
```

### 3.3 ヘッダー・セキュリティ設定

```javascript
// next.config.js
export default {
  async headers() {
    return [
      {
        source: '/(.*)',
        headers: [
          {
            key: 'X-Content-Type-Options',
            value: 'nosniff',
          },
          {
            key: 'X-Frame-Options',
            value: 'DENY',
          },
          {
            key: 'X-XSS-Protection',
            value: '1; mode=block',
          },
          {
            key: 'Strict-Transport-Security',
            value: 'max-age=31536000; includeSubDomains',
          },
          {
            key: 'Content-Security-Policy',
            value: "default-src 'self'; script-src 'self' 'unsafe-inline'",
          },
        ],
      },
    ];
  },
};
```

### 3.4 IP ホワイトリスト (Pro/Enterprise)

```
Project > Settings > Security
```

許可する IP アドレスを指定。

### 3.5 監査ログ

```
Team Settings > Audit > Logs
```

アクションの履歴確認：
- デプロイメント
- メンバー追加・削除
- 環境変数変更

## 4. バックアップ・データ保護

### 4.1 Vercel Postgres バックアップ

**自動バックアップ:**

```
Project > Storage > [Database] > Backup
```

- 日次バックアップ (無料)
- 7 日間保持

### 4.2 手動バックアップ

```bash
# データベース全体をダンプ
pg_dump $POSTGRES_URL > backup.sql

# 圧縮
gzip backup.sql

# S3 などにアップロード
aws s3 cp backup.sql.gz s3://my-backups/

# または リカバリ
psql $POSTGRES_URL < backup.sql
```

### 4.3 リポジトリ設定バックアップ

デプロイ設定・環境変数をコード化：

```javascript
// vercel.json
{
  "version": 2,
  "public": false,
  "regions": ["sfo1", "sfo1"],
  "functions": {
    "api/**": {
      "memory": 1024,
      "maxDuration": 60
    }
  },
  "env": {
    "NODE_ENV": "production"
  }
}
```

Git に記録しておく。

### 4.4 ディザスタリカバリープラン

```
目標 RTO: 1時間 (Recovery Time Objective)
目標 RPO: 24時間 (Recovery Point Objective)

フェーズ:
1. インシデント検知 (5分以内)
2. 根本原因調査 (15分以内)
3. ロールバック実行 (10分)
4. 検証・本番環境確認 (15分)
5. 状況報告 (5分)
```

## 5. チーム管理・権限

### 5.1 チームメンバー追加

```
Team Settings > Members > Add
```

### 5.2 ロール・権限

| ロール | 説明 |
|---|---|
| **Owner** | すべての権限 |
| **Admin** | メンバー管理・設定変更 |
| **Developer** | デプロイメント・コード変更 |
| **Viewer** | 読み取り専用 |

### 5.3 プロジェクトレベルの権限

```
Project > Settings > Members > Project Roles
```

プロジェクトごとに権限指定。

### 5.4 OAuth アプリケーション

サードパーティツール連携の際に使用。

```
Account Settings > Apps > Create App
```

## 6. 費用管理

### 6.1 プラン別比較

| | Free | Pro | Enterprise |
|---|---|---|---|
| **計算** | 無制限 | $20/月 | カスタム |
| **バンド幅** | 100GB/月 | 無制限 | 無制限 |
| **並行デプロイ** | 1 | 無制限 | 無制限 |
| **Postgres** | - | 従量課金 | 個別契約 |
| **KV** | - | 従量課金 | 個別契約 |
| **Logs** | 24時間 | 7日間 | 90日間 |

### 6.2 使用状況の確認

```
Team Settings > Usage
```

- 計算時間
- バンド幅
- ストレージ
- データベース

### 6.3 コスト削減

```
1. 不要なデプロイメント削除
   Project > Deployments > [古い] > Delete

2. 画像最適化（転送量削減）
   Next.js Image コンポーネント使用

3. ISR / キャッシング活用
   再ビルド頻度削減

4. Edge Middleware の活用
   サーバーレス関数呼び出し削減

5. Serverless Function 時間短縮
   ロジック最適化
```

## 7. トラブル対応フロー

### 7.1 インシデント対応プロセス

```
1. 検知・検証 (5分)
   - ユーザー報告 / 監視アラート
   - 問題範囲確認
   - 優先度判定 (P1-P4)

2. 一次対応 (15分)
   - ホットスタンバイへの切り替え
   - キャッシュクリア
   - ロールバック検討

3. 詳細調査 (30-60分)
   - ログ確認
   - メトリクス分析
   - 根本原因特定

4. 本修正 (1-24時間)
   - ホットフィックス PR
   - テスト実行
   - デプロイ

5. ポストモーテム (翌営業日)
   - 原因分析
   - 改善案検討
   - 予防策実装
```

### 7.2 通知・エスカレーション

```
P1 (Critical)
├─ Slack: @channel (即座)
├─ Email: team@
└─ Call: 担当者

P2 (High)
├─ Slack: #incidents
└─ Email: team@

P3 (Medium)
└─ Slack: #incidents

P4 (Low)
└─ JIRA チケット作成
```

## 8. 定期メンテナンス

### 8.1 日次チェック

```bash
# 毎朝 9:00
□ Analytics 確認（エラー率・パフォーマンス）
□ ログエラー確認
□ デプロイ成功状況
□ アラート確認
```

### 8.2 週次メンテナンス

```bash
# 毎週月曜 10:00
□ npm audit (セキュリティ)
□ npm outdated (バージョン)
□ Core Web Vitals スコア確認
□ 使用状況・費用確認
□ バックアップ確認
```

### 8.3 月次メンテナンス

```bash
# 毎月 1 日
□ Node.js LTS バージョン確認
□ Next.js メジャーアップデート確認
□ セキュリティアップデート適用
□ データベースバージョン確認
□ ディザスタリカバリー訓練
□ 費用レポート作成
```

### 8.4 四半期ごと

```bash
# 毎 Q1/Q2/Q3/Q4 開始時
□ アーキテクチャレビュー
□ パフォーマンス最適化
□ セキュリティ監査
□ 容量計画レビュー
□ チーム研修（新機能）
```
