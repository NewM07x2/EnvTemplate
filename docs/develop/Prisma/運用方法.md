### 本番運用の原則

| 原則 | 説明 |
|------|------|
| **シングルトンパターン** | PrismaClient のインスタンスは 1 つだけ |
| **コネクションプール** | DB 接続数を適切に管理 |
| **グレースフルシャットダウン** | アプリ終了時に接続を正しく解放 |
| **ログレベルの分離** | 開発 vs 本番でログ出力を変える |
| **エラーの適切なハンドリング** | Prisma エラーをユーザーフレンドリーに変換 |

## 2. コネクションプール管理

### 基本設定

```typescript
// src/lib/prisma/client.ts（本番向け）
import { PrismaClient } from '@prisma/client'

const globalForPrisma = globalThis as unknown as {
  prisma: PrismaClient | undefined
}

export const prisma =
  globalForPrisma.prisma ??
  new PrismaClient({
    log:
      process.env.NODE_ENV === 'development'
        ? ['query', 'error', 'warn']
        : ['error'],
  })

if (process.env.NODE_ENV !== 'production') {
  globalForPrisma.prisma = prisma
}
```

### コネクションプールサイズの設定

```env
# .env
# connection_limit: 同時接続数（デフォルト: num_cpus * 2 + 1）
# pool_timeout: 接続待ちのタイムアウト（秒）
DATABASE_URL="postgresql://user:password@host:5432/mydb?connection_limit=10&pool_timeout=10"
```

### コネクションプールサイズの目安

| 環境 | 推奨値 | 根拠 |
|------|--------|------|
| 開発環境 | 2-5 | ローカルなので少なくてOK |
| 小規模本番（1-2台） | 5-10 | DB の max_connections の 1/4 程度 |
| 中規模本番（3-10台） | 3-5 | サーバー台数 × pool = DB 側の接続数 |
| サーバーレス（Vercel等） | 1-3 | 関数ごとにプールが作られるため少なめ |

### 計算式

```
推奨接続数 = DB の max_connections / アプリサーバーの台数 - 予備(10%)

例: PostgreSQL max_connections = 100, アプリ 4台の場合
  100 / 4 - 10% = 約 22 → connection_limit=20
```

### グレースフルシャットダウン

```typescript
// アプリケーション終了時に接続を解放
process.on('SIGINT', async () => {
  console.log('Graceful shutdown: disconnecting Prisma...')
  await prisma.$disconnect()
  process.exit(0)
})

process.on('SIGTERM', async () => {
  console.log('Graceful shutdown: disconnecting Prisma...')
  await prisma.$disconnect()

})
```

## 4. パフォーマンス最適化

### select で必要なフィールドのみ取得

```typescript
// ❌ Bad: 全フィールドを取得
const users = await prisma.user.findMany()

// ✅ Good: 必要なフィールドのみ取得
const users = await prisma.user.findMany({
  select: {
    id: true,
    name: true,
    email: true,
  },
})
```

### N+1 問題の回避

```typescript
// ❌ Bad: N+1 問題
const users = await prisma.user.findMany()
for (const user of users) {
  const posts = await prisma.post.findMany({
    where: { authorId: user.id },
  })
}

// ✅ Good: include で一括取得（JOINクエリ）
const users = await prisma.user.findMany({
  include: {
    posts: true,
  },
})

// ✅ Good: 必要なフィールドだけ select
const users = await prisma.user.findMany({
  select: {
    id: true,
    name: true,
    posts: {
      select: {
        id: true,
        title: true,
      },
    },
  },
})
```

### バッチ処理の最適化

```typescript
// ❌ Bad: ループで1件ずつ処理
for (const email of emails) {
  await prisma.user.create({ data: { email, name: '', age: 0 } })
}

// ✅ Good: createMany で一括処理
await prisma.user.createMany({
  data: emails.map(email => ({ email, name: '', age: 0 })),
  skipDuplicates: true,
})

// ✅ Good: 大量データの場合はバッチに分割
const BATCH_SIZE = 1000
for (let i = 0; i < allData.length; i += BATCH_SIZE) {
  const batch = allData.slice(i, i + BATCH_SIZE)
  await prisma.user.createMany({
    data: batch,
    skipDuplicates: true,
  })
}
```

### インデックス設計のガイドライン

```prisma
model User {
  id        Int      @id @default(autoincrement())
  email     String   @unique          // ← ユニーク制約 = 自動的にインデックス
  name      String?
  role      Role     @default(USER)
  createdAt DateTime @default(now())
  deleted   Boolean  @default(false)

  // ★ よく使う検索条件にインデックスを追加
  @@index([email])                    // メール検索
  @@index([role, deleted])            // ロール + 削除フラグの複合検索
  @@index([createdAt])                // 日付順ソート
  @@index([deleted, createdAt])       // アクティブユーザーの日付順

  // ★ 複合ユニーク制約
  @@unique([email, deleted])          // 論理削除時の重複許可
}
```

| インデックスのルール | 説明 |
|-------------------|------|
| WHERE 句に頻出するカラム | インデックスを追加 |
| ORDER BY に使うカラム | インデックスを追加 |
| 外部キー | 自動的にインデックスが作られないので明示的に追加 |
| 複合インデックスの順序 | 絞り込みが強いカラムを先に |
| 追加しすぎない | INSERT/UPDATE のパフォーマンスが低下 |

## 5. エラーハンドリング

### Prisma エラーの分類

```typescript
import { Prisma } from '@prisma/client'

// ★ アプリ全体で使えるエラーハンドラー
function handlePrismaError(error: unknown): {
  status: number
  message: string
} {
  // Prisma の既知エラー
  if (error instanceof Prisma.PrismaClientKnownRequestError) {
    switch (error.code) {
      case 'P2002': {
        const target = (error.meta?.target as string[])?.join(', ')
        return {
          status: 409,
          message: `${target} は既に使用されています`,
        }
      }
      case 'P2003':
        return {
          status: 400,
          message: '参照先のデータが存在しません',
        }
      case 'P2025':
        return {
          status: 404,
          message: '対象のデータが見つかりません',
        }
      default:
        return {
          status: 500,
          message: `データベースエラー (${error.code})`,
        }
    }
  }

  // バリデーションエラー
  if (error instanceof Prisma.PrismaClientValidationError) {
    return {
      status: 400,
      message: 'リクエストデータの形式が正しくありません',
    }
  }

  // 接続エラー
  if (error instanceof Prisma.PrismaClientInitializationError) {
    console.error('DB接続エラー:', error)
    return {
      status: 503,
      message: 'データベースに接続できません',
    }
  }

  // タイムアウト
  if (error instanceof Prisma.PrismaClientRustPanicError) {
    console.error('Prisma内部エラー:', error)
    return {
      status: 500,
      message: '内部エラーが発生しました',
    }
  }

  // 予期しないエラー
  console.error('不明なエラー:', error)
  return {
    status: 500,
    message: '予期しないエラーが発生しました',
  }
}
```

### API Route での使用例

```typescript
// src/app/api/users/route.ts
import { NextRequest, NextResponse } from 'next/server'
import { prisma } from '@/lib/prisma/client'

export async function POST(request: NextRequest) {
  try {
    const body = await request.json()
    const user = await prisma.user.create({ data: body })
    return NextResponse.json(user, { status: 201 })
  } catch (error) {
    const { status, message } = handlePrismaError(error)
    return NextResponse.json({ error: message }, { status })
  }
}
```

### エラーコード一覧（主要なもの）

| コード | カテゴリ | 説明 |
|-------|---------|------|
| P2000 | 値エラー | カラムの値が長すぎる |
| P2001 | 検索 | WHERE 条件に一致するレコードがない |
| P2002 | ユニーク | ユニーク制約違反 |
| P2003 | 外部キー | 外部キー制約違反 |
| P2010 | SQL | Raw query が失敗 |
| P2021 | テーブル | テーブルが存在しない |
| P2022 | カラム | カラムが存在しない |
| P2024 | 接続 | 接続プールタイムアウト |
| P2025 | レコード | 操作対象のレコードが見つからない |
| P2028 | トランザクション | トランザクション API エラー |

## 7. バックアップとリストア

### PostgreSQL のバックアップ

```bash
# ━━━ 論理バックアップ（pg_dump） ━━━
# 全データベースのバックアップ
pg_dump -h localhost -U postgres -d mydb -F c -f backup_$(date +%Y%m%d).dump

# スキーマのみバックアップ
pg_dump -h localhost -U postgres -d mydb --schema-only -f schema_$(date +%Y%m%d).sql

# データのみバックアップ
pg_dump -h localhost -U postgres -d mydb --data-only -f data_$(date +%Y%m%d).sql

# 特定テーブルのみ
pg_dump -h localhost -U postgres -d mydb -t "User" -t "Post" -f partial_backup.dump
```

### リストア

```bash
# ダンプファイルからリストア
pg_restore -h localhost -U postgres -d mydb backup_20250315.dump

# SQL ファイルからリストア
psql -h localhost -U postgres -d mydb < schema_20250315.sql
```

### バックアップの自動化（Docker 環境）

```yaml
# docker-compose.yml に追加
services:
  db-backup:
    image: postgres:16-alpine
    depends_on:
      - db
    environment:
      PGHOST: db
      PGUSER: postgres
      PGPASSWORD: postgres
      PGDATABASE: mydb
    volumes:
      - ./backups:/backups
    command: >
      sh -c "while true; do
        pg_dump -F c -f /backups/backup_$$(date +%Y%m%d_%H%M%S).dump;
        echo 'Backup completed at $$(date)';
        sleep 86400;
      done"
```

### マイグレーション前のバックアップ（推奨運用）

```bash

echo "📦 バックアップを作成中..."
```

## 8. ヘルスチェック

### DB 接続のヘルスチェック

```typescript
// src/lib/prisma/healthcheck.ts
import { prisma } from './client'

export async function checkDatabaseHealth(): Promise<{
  status: 'healthy' | 'unhealthy'
  responseTime: number
  message?: string
}> {
  const start = Date.now()

  try {
    // シンプルなクエリで接続確認
    await prisma.$queryRaw`SELECT 1`

    return {
      status: 'healthy',
      responseTime: Date.now() - start,
    }
  } catch (error) {
    return {
      status: 'unhealthy',
      responseTime: Date.now() - start,
      message: error instanceof Error ? error.message : 'Unknown error',
    }
  }
}
```

### API Route でのヘルスチェック

## 9. Prisma Client Extensions

Prisma v4.16+ で利用可能な拡張機能です。

### 論理削除のミドルウェア

```typescript
// src/lib/prisma/client.ts
import { PrismaClient } from '@prisma/client'

const basePrisma = new PrismaClient()

export const prisma = basePrisma.$extends({
  query: {
    user: {
      // findMany 時に自動的に deleted: false を付与
      async findMany({ args, query }) {
        if (args.where?.deleted === undefined) {
          args.where = { ...args.where, deleted: false }
        }
        return query(args)
      },
      // delete を論理削除に変換
      async delete({ args, query }) {
        return basePrisma.user.update({
          where: args.where,
          data: { deleted: true },
        })
      },
    },
  },
})
```


## 10. デプロイ戦略

### ゼロダウンタイムデプロイ

データベースのマイグレーションとアプリケーションのデプロイを安全に行う手順です。

```
フェーズ 1: 後方互換マイグレーション
  ├── 新カラムは nullable で追加
  ├── 旧カラムはまだ削除しない
  └── migrate deploy で適用

フェーズ 2: アプリケーションデプロイ
  ├── 新しいコードが新カラムを使用
  ├── 旧カラムからデータを移行
  └── 新旧両方をサポート

フェーズ 3: クリーンアップマイグレーション
  ├── 旧カラムを削除
  └── NOT NULL 制約を追加
```

### 具体例: カラム名の変更

```
❌ 一度にやると危険:
  RENAME COLUMN name → display_name
  → デプロイ中に旧コードが name を参照してエラー

✅ 安全な手順:
  Step 1: display_name カラムを追加（nullable）
  Step 2: name の値を display_name にコピー
  Step 3: アプリを display_name を使うようにデプロイ
  Step 4: name カラムを削除
```

### Vercel でのデプロイ

## 11. スケーリング

### サーバーレス環境での注意点

サーバーレス環境（Vercel Functions, AWS Lambda 等）では、各関数インスタンスが独立したコネクションプールを持つため、接続数が爆発する可能性があります。

```env
# サーバーレス環境では接続数を少なく
DATABASE_URL="postgresql://user:password@host:5432/mydb?connection_limit=1"
```

### Prisma Accelerate（推奨）

Prisma が提供するコネクションプーリングサービスです。

```env
# Prisma Accelerate を使用
DATABASE_URL="prisma://accelerate.prisma-data.net/?api_key=YOUR_API_KEY"

# 直接の DB URL（マイグレーション用）
DIRECT_URL="postgresql://user:password@host:5432/mydb"
```

```prisma
// schema.prisma
datasource db {
  provider  = "postgresql"
  url       = env("DATABASE_URL")
  directUrl = env("DIRECT_URL")  // マイグレーション用の直接接続
}
```

### リードレプリカの活用

```typescript
// 読み取り専用クエリはリードレプリカに振り分け
const readPrisma = new PrismaClient({
  datasources: {
    db: { url: process.env.READ_DATABASE_URL },
  },
})

const writePrisma = new PrismaClient({
  datasources: {
    db: { url: process.env.WRITE_DATABASE_URL },
  },
})

// 使い分け
const users = await readPrisma.user.findMany()  // 読み取り → レプリカ
await writePrisma.user.create({ data: {...} })   // 書き込み → プライマリ
```

## 12. 監視とアラート

### 監視すべきメトリクス

| メトリクス | 閾値の目安 | アラート |
|-----------|----------|---------|
| クエリ応答時間 | > 500ms | ⚠️ Warning |
| クエリ応答時間 | > 2000ms | 🔴 Critical |
| コネクションプール使用率 | > 80% | ⚠️ Warning |
| エラー率 | > 1% | ⚠️ Warning |
| エラー率 | > 5% | 🔴 Critical |
| DB 接続失敗 | 1回でも | 🔴 Critical |

### カスタムメトリクスの収集

```typescript
import { PrismaClient, Prisma } from '@prisma/client'

// メトリクス収集用のクラス
class PrismaMetrics {
  private queryCount = 0
  private errorCount = 0
  private totalDuration = 0

  attach(prisma: PrismaClient) {
    prisma.$on('query' as never, (e: Prisma.QueryEvent) => {
      this.queryCount++
      this.totalDuration += e.duration

      // スロークエリの検出
      if (e.duration > 500) {
        console.warn(`[SLOW QUERY] ${e.duration}ms: ${e.query}`)
      }
    })
  }

  getMetrics() {
    return {
      queryCount: this.queryCount,
      errorCount: this.errorCount,
      avgDuration: this.queryCount > 0
        ? this.totalDuration / this.queryCount
        : 0,
    }
  }

  reset() {
    this.queryCount = 0
    this.errorCount = 0
    this.totalDuration = 0
  }
}

export const metrics = new PrismaMetrics()
```

## 13. 定期メンテナンス

### PostgreSQL のメンテナンスタスク

```sql
-- テーブルの統計情報を更新（クエリプランの最適化）
ANALYZE;

-- 不要領域を回収（通常は autovacuum が自動実行）
VACUUM ANALYZE;

-- インデックスの再構築（パフォーマンスが劣化した場合）
REINDEX TABLE "User";

-- テーブルサイズの確認
SELECT
  schemaname,
  tablename,
  pg_size_pretty(pg_total_relation_size(schemaname || '.' || tablename)) as total_size,
  pg_size_pretty(pg_relation_size(schemaname || '.' || tablename)) as table_size,
  pg_size_pretty(pg_indexes_size(schemaname || '.' || tablename::regclass)) as index_size
FROM pg_tables
WHERE schemaname = 'public'
ORDER BY pg_total_relation_size(schemaname || '.' || tablename) DESC;
```

### 定期実行スクリプト

```typescript
// scripts/maintenance.ts
import { prisma } from '../src/lib/prisma/client'

async function maintenance() {
  console.log('🔧 定期メンテナンス開始...')

  // 1. 古い論理削除レコードの物理削除（30日以上前）
  const thirtyDaysAgo = new Date()
  thirtyDaysAgo.setDate(thirtyDaysAgo.getDate() - 30)

  const deletedUsers = await prisma.user.deleteMany({
    where: {
      deleted: true,
      updatedAt: { lt: thirtyDaysAgo },
    },
  })
  console.log(`  ✅ 古い削除済みユーザー: ${deletedUsers.count} 件を物理削除`)

  // 2. テーブルの VACUUM ANALYZE
  await prisma.$executeRaw`VACUUM ANALYZE "User"`
  await prisma.$executeRaw`VACUUM ANALYZE "Post"`
  console.log('  ✅ VACUUM ANALYZE 完了')

  // 3. 統計情報の出力
  const userCount = await prisma.user.count()
  const postCount = await prisma.post.count()
  console.log(`  📊 ユーザー数: ${userCount}, 投稿数: ${postCount}`)

  console.log('🔧 定期メンテナンス完了')
}

maintenance()
  .catch(console.error)
  .finally(() => prisma.$disconnect())
```
