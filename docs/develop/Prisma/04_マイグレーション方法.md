## 1. マイグレーションとは

マイグレーションとは、**データベースのスキーマ（構造）をバージョン管理し、安全に変更を適用する仕組み**です。

### マイグレーションの役割

```
┌─────────────────────────────────────────────────────────┐
│         Prisma マイグレーションのライフサイクル           │
│                                                         │
│  schema.prisma        migration SQL         Database    │
│  (スキーマ定義)  →  (変更差分の SQL)  →  (実際の DB)    │
│                                                         │
│  1. スキーマを変更                                        │
│  2. migrate dev で差分SQL生成                             │
│  3. SQL ファイルを Git で管理                              │
│  4. 本番では migrate deploy で適用                        │
└─────────────────────────────────────────────────────────┘
```

### 主要コマンドの比較

| コマンド | 用途 | 環境 | マイグレーション履歴 | データ |
|---------|------|------|-------------------|--------|
| `prisma migrate dev` | 開発中のスキーマ変更 | 開発 | ✅ 記録される | ⚠️ リセットされる場合あり |
| `prisma migrate deploy` | 本番へのデプロイ | 本番/ステージング | ✅ 記録される | ✅ 保持される |
| `prisma migrate reset` | DB の完全リセット | 開発 | ✅ 最初から再適用 | ❌ 全削除 |
| `prisma db push` | プロトタイピング | 開発 | ❌ 記録されない | ⚠️ リセットされる場合あり |
| `prisma db pull` | 既存 DB からスキーマ生成 | どちらも | ❌ 記録されない | ✅ 変更なし |

## 2. マイグレーションの基本フロー

### 開発時の典型的なフロー

```
1. schema.prisma を編集
      ↓
2. npx prisma migrate dev --name 変更名
      ↓
3. マイグレーション SQL が自動生成される
      ↓
4. Prisma Client が自動再生成される
      ↓
5. アプリケーションで動作確認
      ↓
6. Git にコミット（schema.prisma + マイグレーションファイル）
```

### デプロイ時のフロー

```
1. CI/CD パイプラインで main ブランチにマージ
      ↓
2. npx prisma migrate deploy（未適用のマイグレーションを順番に適用）
      ↓
3. アプリケーションをデプロイ
```

## 3. 開発環境のマイグレーション（migrate dev）

`migrate dev` は**開発環境専用**のコマンドです。以下の処理を自動で行います:

1. スキーマの差分を検出
2. マイグレーション SQL を生成
3. DB に SQL を適用
4. `prisma generate` を実行（Client を再生成）

### 基本的な使い方

```bash
# マイグレーションを作成して適用
npx prisma migrate dev --name init

# マイグレーション名の例
npx prisma migrate dev --name add_user_table
npx prisma migrate dev --name add_email_index
npx prisma migrate dev --name add_post_model
npx prisma migrate dev --name change_age_to_optional
```

### オプション

```bash
# SQL ファイルだけ生成（DB には適用しない）
npx prisma migrate dev --create-only

# 名前を付けずに実行（対話的に名前を入力）
npx prisma migrate dev
```

### 動作の詳細

```bash
$ npx prisma migrate dev --name add_post_model

# 出力例:
# Prisma schema loaded from prisma/schema.prisma
# Datasource "db": PostgreSQL database "myapp", schema "public" at "localhost:5432"
#
# Applying migration `20250315120000_add_post_model`
#
# The following migration(s) have been created and applied:
#
# migrations/
#   └── 20250315120000_add_post_model/
#       └── migration.sql
#
# Your database is now in sync with your schema.
# ✔ Generated Prisma Client to ./node_modules/@prisma/client
```

### ⚠️ データ損失の警告が出た場合

スキーマの変更によってデータが失われる可能性がある場合、Prisma は警告を出します。

```
⚠️ We found changes that cannot be executed:
  - Changed the column `age` on the `User` table from `Int` to `String`.

? Are you sure you want to create and apply this migration? (y/N)
```

この場合の選択肢:
- **y**: そのまま適用（データが変換される or 失われる）
- **N**: キャンセルして手動で対処

## 4. 本番環境のマイグレーション（migrate deploy）

`migrate deploy` は**本番環境やステージング環境**で使用するコマンドです。

### 特徴

| 項目 | migrate dev | migrate deploy |
|------|------------|----------------|
| 新しいマイグレーション生成 | ✅ する | ❌ しない |
| 既存マイグレーションの適用 | ✅ する | ✅ する |
| スキーマドリフト検出 | ✅ する | ❌ しない |
| 対話的なプロンプト | ✅ あり | ❌ なし（CI/CD 向け） |
| DB リセット | ⚠️ 必要な場合あり | ❌ 絶対にしない |
| Prisma Client 再生成 | ✅ 自動 | ❌ しない（要手動） |

### 使い方

```bash
# 本番 DB に未適用のマイグレーションを適用
npx prisma migrate deploy
```

### CI/CD での使用例

```yaml
# GitHub Actions
name: Deploy
on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci

      # 1. Prisma Client を生成
      - run: npx prisma generate

      # 2. マイグレーションを適用
      - run: npx prisma migrate deploy
        env:
          DATABASE_URL: ${{ secrets.DATABASE_URL }}

      # 3. アプリケーションをデプロイ
      - run: npm run build
```

### Dockerfile での使用例

```dockerfile
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
COPY prisma ./prisma/
RUN npm ci
RUN npx prisma generate

COPY . .
RUN npm run build

FROM node:20-alpine AS runner
WORKDIR /app
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/prisma ./prisma
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/package.json .

# コンテナ起動時にマイグレーションを適用してからアプリ起動
CMD ["sh", "-c", "npx prisma migrate deploy && node dist/main.js"]
```

## 5. マイグレーションのリセット（migrate reset）

**開発環境で DB を初期状態に戻す**ためのコマンドです。

### 動作内容

1. データベースを完全に削除（DROP）
2. データベースを再作成
3. すべてのマイグレーションを最初から順番に適用
4. シードスクリプトを実行（設定されている場合）

```bash
# DB をリセット
npx prisma migrate reset

# 確認プロンプトをスキップ
npx prisma migrate reset --force

# シードをスキップ
npx prisma migrate reset --skip-seed
```

### いつ使うか

| シチュエーション | migrate reset が有効 |
|----------------|---------------------|
| マイグレーション履歴が壊れた | ✅ |
| テストデータをクリーンアップしたい | ✅ |
| ドリフトを解消したい | ✅ |
| 本番環境でデータを消したい | ❌ 絶対に使わない |

> **⚠️ 警告:** `migrate reset` は**全データが削除**されます。本番環境では絶対に実行しないでください。

## 6. スキーマの直接反映（db push）

`db push` はマイグレーションファイルを**生成せずに**スキーマを DB に直接反映します。

```bash
npx prisma db push
```

### 用途

| 用途 | 適切さ |
|------|-------|
| プロトタイピング・試行錯誤 | ✅ 最適 |
| スキーマを頻繁に変更する初期開発 | ✅ 適切 |
| チーム開発 | ⚠️ 非推奨（履歴が残らない） |
| 本番デプロイ | ❌ 使わない |

### migrate dev との違い

```
db push:
  schema.prisma → (直接反映) → Database
  ※ マイグレーションファイルは生成されない
  ※ 履歴が残らない

migrate dev:
  schema.prisma → migration.sql → Database
  ※ マイグレーションファイルが生成される
  ※ 履歴が Git で管理できる
```

### 典型的なワークフロー

```
初期開発（プロトタイプ）:
  db push で素早く試行錯誤
      ↓
開発が安定してきたら:
  migrate dev に移行して履歴管理を開始
      ↓
本番デプロイ:
  migrate deploy で確実に適用
```

## 7. DB からスキーマを逆生成（db pull）

既存のデータベースから `schema.prisma` を**自動生成（イントロスペクション）**します。

```bash
npx prisma db pull
```

### 用途

- 既存プロジェクトに Prisma を導入する場合
- 手動で DB を変更した後にスキーマを同期する場合
- DB の現在の状態を確認する場合

### ワークフロー: 既存 DB に Prisma を導入

```bash
# 1. Prisma を初期化
npx prisma init

# 2. DATABASE_URL を .env に設定

# 3. 既存 DB からスキーマを生成
npx prisma db pull

# 4. 生成されたスキーマを確認・修正
cat prisma/schema.prisma

# 5. Prisma Client を生成
npx prisma generate

# 6. 以降は migrate dev で管理
npx prisma migrate dev --name baseline
```

## 8. マイグレーションファイルの構造

### ディレクトリ構造

```
prisma/
├── schema.prisma
└── migrations/
    ├── 20250310134031_init/
    │   └── migration.sql
    ├── 20250315120000_add_post_model/
    │   └── migration.sql
    ├── 20250320090000_add_tags/
    │   └── migration.sql
    └── migration_lock.toml      ← DB プロバイダのロックファイル
```

### migration.sql の内容例

```sql
-- prisma/migrations/20250310134031_init/migration.sql
-- CreateTable
CREATE TABLE "User" (
    "id" SERIAL NOT NULL,
    "email" TEXT NOT NULL,
    "name" TEXT,
    "age" INTEGER NOT NULL,
    "role" TEXT NOT NULL DEFAULT 'USER',
    "createdAt" TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,
    "updatedAt" TIMESTAMP(3) NOT NULL,
    "deleted" BOOLEAN NOT NULL DEFAULT false,

    CONSTRAINT "User_pkey" PRIMARY KEY ("id")
);

-- CreateIndex
CREATE UNIQUE INDEX "User_email_key" ON "User"("email");

-- CreateIndex
CREATE INDEX "User_email_idx" ON "User"("email");
CREATE INDEX "User_createdAt_idx" ON "User"("createdAt");
```

### migration_lock.toml

```toml
# Please do not edit this file manually
# It should be added in your version-control system (i.e. Git)
provider = "postgresql"
```

> **重要:** `migration_lock.toml` は必ず Git にコミットしてください。DB プロバイダの不一致を防ぎます。

## 9. スキーマ変更の具体例

### ケース 1: カラムの追加

```prisma
model User {
  // ... 既存フィールド
  phoneNumber String?    // ← 新規追加（nullable にすれば安全）
}
```

```bash
npx prisma migrate dev --name add_phone_number
```

生成される SQL:
```sql
ALTER TABLE "User" ADD COLUMN "phoneNumber" TEXT;
```

### ケース 2: カラムの型変更

```prisma
model User {
  age String   // Int → String に変更
}
```

```bash
npx prisma migrate dev --name change_age_type
```

> **⚠️ 注意:** 既存データの変換が必要。データ損失の警告が出る場合があります。

### ケース 3: 新しいテーブルの追加

```prisma
model Comment {
  id        Int      @id @default(autoincrement())
  text      String
  postId    Int
  post      Post     @relation(fields: [postId], references: [id], onDelete: Cascade)
  createdAt DateTime @default(now())
}

model Post {
  // ... 既存フィールド
  comments Comment[]  // ← リレーション追加
}
```

```bash
npx prisma migrate dev --name add_comment_model
```

### ケース 4: インデックスの追加

```prisma
model User {
  // ... 既存フィールド

  @@index([name])                    // 単一カラムインデックス
  @@index([role, createdAt])         // 複合インデックス
  @@unique([email, deleted])         // 複合ユニーク制約
}
```

```bash
npx prisma migrate dev --name add_indexes
```

### ケース 5: カラムの削除

```prisma
model User {
  // age Int  ← この行を削除
}
```

```bash
npx prisma migrate dev --name remove_age_column
```

> **⚠️ 注意:** カラムの削除は**データが完全に失われます**。本番では事前にデータバックアップを取得してください。

### ケース 6: 列挙型の値追加

```prisma
enum Role {
  USER
  ADMIN
  EDITOR
  MODERATOR   // ← 新規追加
}
```

```bash
npx prisma migrate dev --name add_moderator_role
```

## 10. マイグレーションの修正・カスタマイズ

### 生成された SQL をカスタマイズする

```bash
# 1. SQL ファイルだけ生成（適用しない）
npx prisma migrate dev --create-only --name add_default_data

# 2. 生成された migration.sql を手動で編集
# 例: デフォルトデータの挿入を追加
```

```sql
-- prisma/migrations/20250320090000_add_default_data/migration.sql

-- Prisma が自動生成した SQL
ALTER TABLE "User" ADD COLUMN "role" TEXT NOT NULL DEFAULT 'USER';

-- ★ 手動で追加: 既存ユーザーのロールを設定
UPDATE "User" SET "role" = 'ADMIN' WHERE "email" = 'admin@example.com';

-- ★ 手動で追加: マスターデータの投入
INSERT INTO "Tag" ("name") VALUES
  ('TypeScript'),
  ('React'),
  ('Next.js'),
  ('Prisma');
```

```bash
# 3. 編集後に適用
npx prisma migrate dev
```

### NOT NULL カラムを追加する場合のパターン

既存テーブルに NOT NULL カラムを追加する場合、既存レコードのデフォルト値が必要です。

```bash
# 1. SQL だけ生成
npx prisma migrate dev --create-only --name add_required_bio
```

```sql
-- 自動生成された SQL を修正
-- ❌ これだとエラー（既存レコードの bio が NULL）
-- ALTER TABLE "User" ADD COLUMN "bio" TEXT NOT NULL;

-- ✅ ステップ 1: nullable で追加
ALTER TABLE "User" ADD COLUMN "bio" TEXT;

-- ✅ ステップ 2: 既存レコードにデフォルト値を設定
UPDATE "User" SET "bio" = '未設定' WHERE "bio" IS NULL;

-- ✅ ステップ 3: NOT NULL 制約を追加
ALTER TABLE "User" ALTER COLUMN "bio" SET NOT NULL;
```

```bash
# 2. 適用
npx prisma migrate dev
```

## 11. チーム開発でのワークフロー

### 基本ルール

| ルール | 説明 |
|-------|------|
| ① Git でマイグレーションを管理 | `prisma/migrations/` ディレクトリは必ずコミット |
| ② マイグレーション SQL は編集しない | 適用済みのファイルは改変禁止（履歴の整合性が壊れる） |
| ③ 競合はリセットで解決 | マイグレーション競合時は `migrate reset` |
| ④ feature ブランチで開発 | main ブランチ直接のマイグレーションは避ける |

### ブランチ運用フロー

```
main ブランチ:
  migrations/
    └── 20250301_init/

feature-A ブランチ:                  feature-B ブランチ:
  migrations/                          migrations/
    ├── 20250301_init/                   ├── 20250301_init/
    └── 20250310_add_posts/              └── 20250310_add_comments/

↓ feature-A をマージ

main ブランチ:
  migrations/
    ├── 20250301_init/
    └── 20250310_add_posts/

↓ feature-B をマージ（競合なし: 異なるファイル名）

main ブランチ:
  migrations/
    ├── 20250301_init/
    ├── 20250310_add_posts/
    └── 20250310_add_comments/   ← 両方のマイグレーションが共存
```

### マージ後の操作

```bash
# feature ブランチをマージした後
git pull origin main

# 新しいマイグレーションを適用
npx prisma migrate dev

# もし競合エラーが出たら
npx prisma migrate reset  # 開発 DB をリセット
```

### .gitignore の推奨設定

```gitignore
# .gitignore
node_modules/
.env

# ❌ これは追加しない！マイグレーションは Git 管理する
# prisma/migrations/
```

## 12. ドリフト検出と解決

**ドリフト**とは、`schema.prisma` の定義と実際の DB の状態がずれている状態です。

### ドリフトが発生する原因

| 原因 | 対処法 |
|------|--------|
| DB を直接 SQL で変更した | `db pull` → `migrate dev` |
| マイグレーションファイルを手動削除した | `migrate reset` |
| 別ブランチのマイグレーションを適用後に戻した | `migrate reset` |

### ドリフトの検出

```bash
# スキーマと DB の差分を確認
npx prisma migrate diff \
  --from-schema-datamodel prisma/schema.prisma \
  --to-schema-datasource prisma/schema.prisma
```

### ドリフトの解決手順

```bash
# 方法 1: DB をリセット（開発環境のみ）
npx prisma migrate reset

# 方法 2: 現在の DB 状態をベースラインとして設定
npx prisma migrate resolve --applied 20250310134031_init
```

## 13. トラブルシューティング

### エラー: "Migration failed to apply cleanly to the shadow database"

```
Error: Migration `20250310_xxx` failed to apply cleanly to the shadow database.
```

**原因:** マイグレーション SQL に構文エラーがある、または DB の状態と不整合がある。

**対処法:**
```bash
# 1. シャドウ DB の接続を確認
# 2. マイグレーション SQL の内容を確認
# 3. 必要なら DB をリセット
npx prisma migrate reset
```

### エラー: "The migration ... was modified after it was applied"

```
Error: The migration `20250310_xxx` was modified after it was applied.
```

**原因:** 適用済みのマイグレーションファイルが変更された。

**対処法:**
```bash
# Git で元に戻す
git checkout -- prisma/migrations/20250310_xxx/migration.sql

# または DB をリセット
npx prisma migrate reset
```

### エラー: "Database needs to be reset"

```
Error: The database schema is not empty.
We need to reset the PostgreSQL database.
```

**対処法:**
```bash
# 開発環境なら安全にリセット
npx prisma migrate reset --force
```

### エラー: "P3009 - migrate found failed migrations"

```
Error: P3009 migrate found failed migrations in the target database.
```

**原因:** 前回のマイグレーション適用が途中で失敗した。

**対処法:**
```bash
# 失敗したマイグレーションをロールバック済みとしてマーク
npx prisma migrate resolve --rolled-back 20250310_xxx

# または適用済みとしてマーク（手動で DB を修正した場合）
npx prisma migrate resolve --applied 20250310_xxx
```

### マイグレーション状態の確認

```bash
# 適用済みマイグレーションの一覧
npx prisma migrate status

# 出力例:
```

## 14. コマンドリファレンス

### マイグレーション関連

| コマンド | 説明 | 環境 |
|---------|------|------|
| `npx prisma migrate dev` | マイグレーション作成 + 適用 | 開発 |
| `npx prisma migrate dev --name <名前>` | 名前付きマイグレーション | 開発 |
| `npx prisma migrate dev --create-only` | SQL 生成のみ（適用しない） | 開発 |
| `npx prisma migrate deploy` | 未適用マイグレーションを適用 | 本番/ステージング |
| `npx prisma migrate reset` | DB リセット + 全マイグレーション再適用 | 開発 |
| `npx prisma migrate reset --force` | 確認スキップでリセット | 開発 |
| `npx prisma migrate status` | マイグレーション状態の確認 | どちらも |
| `npx prisma migrate resolve` | マイグレーション状態の手動解決 | どちらも |
| `npx prisma migrate diff` | スキーマ差分の確認 | どちらも |

### データベース関連

| コマンド | 説明 | 環境 |
|---------|------|------|
| `npx prisma db push` | スキーマを直接 DB に反映 | 開発 |
| `npx prisma db pull` | DB からスキーマを逆生成 | どちらも |
| `npx prisma db seed` | シードスクリプトを実行 | 開発 |
| `npx prisma db execute` | 任意の SQL を実行 | どちらも |

### その他

| コマンド | 説明 |
|---------|------|
| `npx prisma generate` | Prisma Client を生成 |
| `npx prisma studio` | GUI 管理画面を起動 |
| `npx prisma validate` | スキーマの構文チェック |
| `npx prisma format` | スキーマのフォーマット |
