## 2. Docker Compose で PostgreSQL を起動

### docker-compose.yml の作成

本リポジトリの各テンプレートには `docker-compose.yml` が用意されています。  
以下は `next/` テンプレートの実際の構成です。

```yaml
# docker-compose.yml
version: "3.8"

services:
  app:
    build:
      context: .
      dockerfile: ./docker/Dockerfile
    ports:
      - "3000:3000"
    volumes:
      - .:/app
      - /app/node_modules
    environment:
      - NODE_ENV=development
      - DATABASE_URL=postgresql://postgres:postgres@db:5432/nextapp?schema=public
    depends_on:
      - db
    command: npm run dev

  db:
    image: postgres:16-alpine
    restart: always
    ports:
      - "5432:5432"
    environment:
      - POSTGRES_USER=postgres
      - POSTGRES_PASSWORD=postgres
      - POSTGRES_DB=nextapp
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:
```

### DB のみ起動する場合

開発中は DB だけ Docker で起動し、アプリはローカルで動かすのが一般的です。

```bash
# DB コンテナのみ起動
docker compose up db -d

# 確認
docker compose ps
# NAME    IMAGE               STATUS    PORTS
# db      postgres:16-alpine  running   0.0.0.0:5432->5432/tcp
```

### 全サービスを起動する場合

```bash
# 全サービスを起動（バックグラウンド）
docker compose up -d

# ログを確認
docker compose logs -f
```

## 4. 環境変数の設定

### .env ファイルの作成

```bash
# .env（プロジェクトルートに作成）
DATABASE_URL="postgresql://postgres:postgres@localhost:5432/nextapp?schema=public"
```

### 接続文字列の構造

```
postgresql://USER:PASSWORD@HOST:PORT/DATABASE?schema=SCHEMA
              │      │       │    │     │           │
              │      │       │    │     │           └─ スキーマ名（通常は public）
              │      │       │    │     └─ データベース名
              │      │       │    └─ ポート番号
              │      │       └─ ホスト名
              │      └─ パスワード
              └─ ユーザー名
```

### Docker 環境とローカル環境の違い

| 環境 | HOST の値 | 理由 |
|------|----------|------|
| **ローカル**（アプリを npm run dev で起動） | `localhost` | ホストマシンから Docker コンテナに接続 |
| **Docker 内**（アプリも Docker で起動） | `db`（サービス名） | Docker ネットワーク内の名前解決 |

```env
# ローカル実行時
DATABASE_URL="postgresql://postgres:postgres@localhost:5432/nextapp?schema=public"

# Docker Compose 内での実行時（docker-compose.yml の environment に記載）
DATABASE_URL="postgresql://postgres:postgres@db:5432/nextapp?schema=public"
```

### .env.example の活用

```bash
# .env.example（Git にコミットするテンプレート）
DATABASE_URL="postgresql://postgres:postgres@localhost:5432/nextapp?schema=public"
```

```bash
# .env.example から .env を作成
cp .env.example .env
```

> **重要:** `.env` ファイルは `.gitignore` に追加し、Git にコミットしないでください。

## 5. Prisma の初期化とマイグレーション

### ステップ 1: DB が起動していることを確認

```bash
# PostgreSQL コンテナの状態を確認
docker compose ps

# DB に接続できるか確認
docker compose exec db psql -U postgres -d nextapp -c "SELECT 1"
```

### ステップ 2: スキーマファイルを確認

本リポジトリのテンプレートには、既にスキーマファイルが用意されています。

**next テンプレートの場合:**

```
next/
└── src/
    └── lib/
        └── prisma/
            ├── schema.prisma   ← スキーマ定義
            ├── client.ts       ← PrismaClient のシングルトン
            └── migrations/     ← マイグレーションファイル
```

**remix テンプレートの場合:**

```
remix/
└── prisma/
    └── schema.prisma   ← スキーマ定義（標準的な配置）
```

### ステップ 3: マイグレーションを実行

```bash
# 初回マイグレーション
npx prisma migrate dev --name init

# 出力例:
# ✔ Generated Prisma Client
# The following migration(s) have been created and applied:
#   migrations/
#     └── 20250310134031_init/
#         └── migration.sql
```

### ステップ 4: Prisma Client の生成を確認

```bash
# 手動で生成（通常は migrate dev 時に自動実行される）
npx prisma generate
```

## 6. シードデータの投入

### シードスクリプトの用意

```typescript
// prisma/seed.ts
import { PrismaClient } from '@prisma/client'

const prisma = new PrismaClient()

async function main() {
  console.log('🌱 シードデータを投入中...')

  // テストユーザーを作成
  const alice = await prisma.user.upsert({
    where: { email: 'alice@example.com' },
    update: {},
    create: {
      email: 'alice@example.com',
      name: 'Alice',
      age: 25,
    },
  })

  const bob = await prisma.user.upsert({
    where: { email: 'bob@example.com' },
    update: {},
    create: {
      email: 'bob@example.com',
      name: 'Bob',
      age: 30,
    },
  })

  console.log('✅ シード完了:', { alice, bob })
}

main()
  .catch((e) => {
    console.error(e)
    process.exit(1)
  })
  .finally(async () => {
    await prisma.$disconnect()
  })
```

### package.json にシード設定を追加

```json
{
  "prisma": {
    "seed": "ts-node --compiler-options {\"module\":\"CommonJS\"} prisma/seed.ts"
  }
}
```

### シードの実行

```bash
# シードスクリプトを実行
npx prisma db seed

# または、migrate reset（リセット + マイグレーション + シード）
npx prisma migrate reset
```

## 9. Docker Compose 構成の詳細解説

### サービス構成図

```
docker-compose.yml
│
├── app サービス（アプリケーション）
│   ├── image: Node.js (Dockerfile でビルド)
│   ├── port: 3000 → 3000
│   ├── volume: ソースコードをマウント
│   ├── env: DATABASE_URL=postgresql://postgres:postgres@db:5432/nextapp
│   └── depends_on: db
│
├── db サービス（PostgreSQL）
│   ├── image: postgres:16-alpine
│   ├── port: 5432 → 5432
│   ├── env: POSTGRES_USER, POSTGRES_PASSWORD, POSTGRES_DB
│   └── volume: postgres_data（永続化）
│
└── volumes
    └── postgres_data（名前付きボリューム）
```

### 各設定項目の説明

```yaml
services:
  db:
    image: postgres:16-alpine
    # ↑ PostgreSQL 16 の Alpine Linux ベースイメージ（軽量）

    restart: always
    # ↑ コンテナが停止しても自動的に再起動

    ports:
      - "5432:5432"
    # ↑ ホストの 5432 ポートをコンテナの 5432 にマッピング
    # ローカルから接続する場合: localhost:5432

    environment:
      - POSTGRES_USER=postgres     # DB ユーザー名
      - POSTGRES_PASSWORD=postgres # DB パスワード
      - POSTGRES_DB=nextapp        # 初期作成されるデータベース名
    # ↑ コンテナ初回起動時に自動的にデータベースが作成される

    volumes:
      - postgres_data:/var/lib/postgresql/data
    # ↑ 名前付きボリュームでデータを永続化
    # コンテナを再作成してもデータが保持される

volumes:
  postgres_data:
  # ↑ Docker が管理する名前付きボリュームの宣言
```

### ポートの変更方法

他のプロジェクトで PostgreSQL を使っている場合、ポートが衝突します。

```yaml
# ポートを変更する例: ホスト側を 5433 に変更
db:
  ports:
    - "5433:5432"  # ホスト:コンテナ
```

```env
# .env も合わせて変更
DATABASE_URL="postgresql://postgres:postgres@localhost:5433/nextapp?schema=public"
```

## 10. VS Code の推奨設定

### settings.json

```json
{
  // Prisma ファイルの自動フォーマット
  "[prisma]": {
    "editor.defaultFormatter": "Prisma.prisma",
    "editor.formatOnSave": true
  }
}
```

### Prisma スキーマのカスタム配置

`schema.prisma` がデフォルトの `prisma/` 以外にある場合（例: `src/lib/prisma/`）:

```json
// package.json
{
  "prisma": {
    "schema": "src/lib/prisma/schema.prisma"
  }
}
```

または、各 Prisma コマンドで `--schema` オプションを指定:

```bash
npx prisma migrate dev --schema src/lib/prisma/schema.prisma
npx prisma generate --schema src/lib/prisma/schema.prisma
npx prisma studio --schema src/lib/prisma/schema.prisma
```

## 11. 環境の起動・停止・リセット

### よく使うコマンド一覧

| 操作 | コマンド |
|------|---------|
| DB 起動 | `docker compose up db -d` |
| 全サービス起動 | `docker compose up -d` |
| 停止 | `docker compose stop` |
| 停止 + コンテナ削除 | `docker compose down` |
| 停止 + コンテナ + ボリューム削除 | `docker compose down -v` |
| DB ログ確認 | `docker compose logs db` |
| DB に接続 | `docker compose exec db psql -U postgres -d nextapp` |
| コンテナ状態確認 | `docker compose ps` |

### データの完全リセット

```bash
# 方法 1: Docker ボリュームごと削除
docker compose down -v
docker compose up db -d
npx prisma migrate dev

# 方法 2: Prisma のリセット機能を使用
npx prisma migrate reset --force
```

### DB に直接接続して操作

```bash
# psql で接続
docker compose exec db psql -U postgres -d nextapp

# SQL を直接実行
nextapp=# SELECT * FROM "User";
nextapp=# \dt          -- テーブル一覧
nextapp=# \d "User"    -- テーブル定義
nextapp=# \q            -- 終了
```

## 12. トラブルシューティング

### ❌ エラー: "Can't reach database server"

```
Error: P1001: Can't reach database server at `localhost:5432`
```

**原因と対処法:**

| 原因 | 対処法 |
|------|--------|
| Docker が起動していない | Docker Desktop を起動 |
| DB コンテナが起動していない | `docker compose up db -d` |
| ポートが間違っている | `.env` の `DATABASE_URL` を確認 |
| 別のサービスがポートを占有 | `lsof -i :5432`（Mac/Linux）で確認 |

```bash
# DB コンテナの状態を確認
docker compose ps

# DB コンテナのログを確認
docker compose logs db

# ポートの使用状況を確認（Windows）
netstat -an | findstr 5432
```

### ❌ エラー: "password authentication failed"

```
Error: P1000: Authentication failed against database server
```

**対処法:**
```bash
# 1. .env のユーザー名とパスワードを確認
cat .env

# 2. docker-compose.yml の環境変数と一致しているか確認
# POSTGRES_USER=postgres
# POSTGRES_PASSWORD=postgres

# 3. ボリュームを削除して再作成（パスワード変更時は必須）
docker compose down -v
docker compose up db -d
```

### ❌ エラー: "database does not exist"

```
Error: P1003: Database `mydb` does not exist on the database server
```

**対処法:**
```bash
# docker-compose.yml の POSTGRES_DB を確認
# .env の DATABASE_URL のデータベース名と一致させる

# 手動でデータベースを作成
docker compose exec db psql -U postgres -c "CREATE DATABASE mydb;"
```

### ❌ エラー: "Prisma schema not found"

```
Error: Could not find a `schema.prisma` file
```

**対処法:**
```bash
# スキーマファイルの場所を確認
find . -name "schema.prisma"

# package.json でカスタムパスを指定
# "prisma": { "schema": "src/lib/prisma/schema.prisma" }
```

### ❌ エラー: ポート 5432 が既に使用されている

```
Error: Bind for 0.0.0.0:5432 failed: port is already allocated
```

**対処法:**

```bash
# 方法 1: 既存のプロセスを停止
# Windows
netstat -ano | findstr 5432
taskkill /PID <PID> /F

# 方法 2: docker-compose.yml でポートを変更
# ports:
#   - "5433:5432"
```

### ❌ エラー: "Migration ... was not found"

マイグレーション履歴と実際のファイルが不一致の場合。

```bash
# 開発環境なら DB をリセット
npx prisma migrate reset --force

# その後マイグレーションを再適用
npx prisma migrate dev
```

### 💡 よくある質問

**Q: Docker を使わずに PostgreSQL を直接インストールしてもよいですか？**

A: 可能ですが、Docker の使用を推奨します。理由:
- チーム全員が同じ環境を再現できる
- バージョンの切り替えが容易
- プロジェクトごとに独立した DB を持てる
- クリーンアップが簡単（`docker compose down -v`）

**Q: SQLite を使いたい場合は？**

A: `schema.prisma` の `datasource` を変更するだけで対応できます。

```prisma
datasource db {
  provider = "sqlite"
  url      = "file:./dev.db"
}
```

```bash
npx prisma migrate dev --name init
# → prisma/dev.db ファイルが自動的に作成される
```

**Q: 複数のプロジェクトで同時に PostgreSQL を使いたい場合は？**

A: プロジェクトごとに異なるポートとデータベース名を使用してください。

```yaml
# プロジェクト A: ポート 5432, DB名: project_a
# プロジェクト B: ポート 5433, DB名: project_b
```
