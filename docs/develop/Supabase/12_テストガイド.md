## 1. テスト戦略の全体像

Supabase アプリのテストは **4 層** で構成します。

```
┌─────────────────────────────────────┐
│  E2E テスト (Playwright)             │  ← ユーザー操作全体を検証
├─────────────────────────────────────┤
│  結合テスト (Vitest + Supabase CLI)  │  ← フロント ↔ DB の連携を検証
├─────────────────────────────────────┤
│  RLS ポリシーテスト (pgTAP)         │  ← 権限制御を検証
├─────────────────────────────────────┤
│  DB ユニットテスト (pgTAP)          │  ← 関数・トリガーを検証
└─────────────────────────────────────┘
```

| テスト種別 | ツール | 検証対象 | 実行速度 |
|-----------|--------|---------|---------|
| DB ユニット | pgTAP | SQL 関数・トリガー・制約 | 速い |
| RLS テスト | pgTAP | Row Level Security ポリシー | 速い |
| 結合テスト | Vitest | フロント SDK ↔ Supabase | 中程度 |
| E2E テスト | Playwright | ブラウザ操作全体 | 遅い |


## 3. pgTAP による DB ユニットテスト

**pgTAP** は PostgreSQL 上で動くテストフレームワークです。SQL 関数・トリガー・制約を直接テストできます。

### セットアップ

```bash
# supabase/tests/ ディレクトリに配置するのが慣例
mkdir -p supabase/tests
```

### テストファイルの書き方

```sql
-- supabase/tests/01_user_functions.test.sql

BEGIN;

-- pgTAP を有効化
SELECT plan(5);  -- テスト数を宣言

-- ① テーブルが存在するか
SELECT has_table('public', 'profiles', 'profiles テーブルが存在する');

-- ② カラムが存在するか
SELECT has_column('public', 'profiles', 'username', 'username カラムが存在する');

-- ③ NOT NULL 制約があるか
SELECT col_not_null('public', 'profiles', 'username', 'username は NOT NULL');

-- ④ 関数の戻り値をテスト
SELECT is(
  (SELECT get_user_display_name('12345678-1234-1234-1234-123456789abc')),
  'テストユーザー',
  'get_user_display_name が正しい表示名を返す'
);

-- ⑤ 外部キー制約のテスト
SELECT fk_ok(
  'public', 'posts', 'user_id',
  'public', 'profiles', 'id',
  'posts.user_id は profiles.id を参照する'
);

SELECT * FROM finish();

ROLLBACK;  -- テスト後はロールバックして DB をクリーンに保つ
```

### テスト実行

```bash
# 全テストファイルを実行
supabase test db

# 特定のファイルだけ実行
supabase test db supabase/tests/01_user_functions.test.sql
```

出力例：
```
ok 1 - profiles テーブルが存在する
ok 2 - username カラムが存在する
ok 3 - username は NOT NULL
ok 4 - get_user_display_name が正しい表示名を返す
ok 5 - posts.user_id は profiles.id を参照する
1..5
```

## 4. RLS ポリシーのテスト

RLS（Row Level Security）のテストは**最重要**です。意図しないデータ漏洩を防ぐために必ずテストしてください。

### RLS テストの書き方

```sql
-- supabase/tests/02_rls_policies.test.sql

BEGIN;

SELECT plan(6);

-- テスト用ユーザー ID を定義
DO $$
BEGIN
  -- user_a と user_b を想定した UUID
  PERFORM set_config('test.user_a', '11111111-1111-1111-1111-111111111111', TRUE);
  PERFORM set_config('test.user_b', '22222222-2222-2222-2222-222222222222', TRUE);
END $$;

-- user_a のデータを挿入（service_role で直接挿入）
INSERT INTO profiles (id, username) VALUES
  ('11111111-1111-1111-1111-111111111111', 'user_a'),
  ('22222222-2222-2222-2222-222222222222', 'user_b');

INSERT INTO posts (id, user_id, title, content) VALUES
  ('aaaaaaaa-0000-0000-0000-000000000000', '11111111-1111-1111-1111-111111111111', 'user_a の投稿', '本文A'),
  ('bbbbbbbb-0000-0000-0000-000000000000', '22222222-2222-2222-2222-222222222222', 'user_b の投稿', '本文B');

-- ① user_a として認証: 自分の投稿が見えるか
SET LOCAL role TO authenticated;
SET LOCAL request.jwt.claims TO '{"sub": "11111111-1111-1111-1111-111111111111", "role": "authenticated"}';

SELECT is(
  (SELECT COUNT(*)::int FROM posts WHERE user_id = '11111111-1111-1111-1111-111111111111'),
  1,
  'user_a は自分の投稿を1件取得できる'
);

-- ② user_a として認証: 他人の投稿が見えないか（RLS が正しく機能しているか）
SELECT is(
  (SELECT COUNT(*)::int FROM posts WHERE user_id = '22222222-2222-2222-2222-222222222222'),
  0,
  'user_a は user_b の投稿を取得できない'
);

-- ③ 未認証ユーザー: 投稿が見えないか
SET LOCAL role TO anon;
RESET request.jwt.claims;

SELECT is(
  (SELECT COUNT(*)::int FROM posts),
  0,
  '未認証ユーザーは投稿を取得できない'
);

-- ④ user_a は自分の投稿を更新できるか
SET LOCAL role TO authenticated;
SET LOCAL request.jwt.claims TO '{"sub": "11111111-1111-1111-1111-111111111111", "role": "authenticated"}';

UPDATE posts SET title = '更新済み' WHERE id = 'aaaaaaaa-0000-0000-0000-000000000000';

SELECT is(
  (SELECT title FROM posts WHERE id = 'aaaaaaaa-0000-0000-0000-000000000000'),
  '更新済み',
  'user_a は自分の投稿を更新できる'
);

-- ⑤ user_a は他人の投稿を更新できないか
UPDATE posts SET title = '不正更新' WHERE id = 'bbbbbbbb-0000-0000-0000-000000000000';

SELECT is(
  (SELECT title FROM posts WHERE id = 'bbbbbbbb-0000-0000-0000-000000000000'),
  'user_b の投稿',  -- 変更されていないことを確認
  'user_a は user_b の投稿を更新できない'
);

-- ⑥ user_a は自分の投稿を削除できるか
DELETE FROM posts WHERE id = 'aaaaaaaa-0000-0000-0000-000000000000';

SELECT is(
  (SELECT COUNT(*)::int FROM posts WHERE id = 'aaaaaaaa-0000-0000-0000-000000000000'),
  0,
  'user_a は自分の投稿を削除できる'
);

SELECT * FROM finish();

ROLLBACK;
```

## 5. Edge Functions のテスト

### Deno テストの基本

```typescript
// supabase/functions/send-email/index_test.ts

import { assertEquals, assertExists } from "https://deno.land/std@0.168.0/testing/asserts.ts";
import { createClient } from "https://esm.sh/@supabase/supabase-js@2";

const SUPABASE_URL = Deno.env.get("SUPABASE_URL") ?? "http://localhost:54321";
const SUPABASE_ANON_KEY = Deno.env.get("SUPABASE_ANON_KEY") ?? "";

Deno.test("send-email function: 正常なリクエスト", async () => {
  const res = await fetch(`${SUPABASE_URL}/functions/v1/send-email`, {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      "Authorization": `Bearer ${SUPABASE_ANON_KEY}`,
    },
    body: JSON.stringify({
      to: "test@example.com",
      subject: "テストメール",
      body: "これはテストです",
    }),
  });

  assertEquals(res.status, 200);
  const data = await res.json();
  assertExists(data.messageId);
});

Deno.test("send-email function: 必須フィールド不足でエラー", async () => {
  const res = await fetch(`${SUPABASE_URL}/functions/v1/send-email`, {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      "Authorization": `Bearer ${SUPABASE_ANON_KEY}`,
    },
    body: JSON.stringify({ to: "test@example.com" }),  // subject が無い
  });

  assertEquals(res.status, 400);
});
```

### Edge Functions のローカルテスト実行

```bash
# 関数をローカルで起動
supabase functions serve send-email --env-file .env.local

# 別ターミナルでテスト実行
deno test --allow-net --allow-env supabase/functions/send-email/index_test.ts
```

```typescript
import { vi } from "vitest";

export const supabase = {
```

### API 関数のユニットテスト

```typescript
// src/lib/posts.test.ts

import { describe, it, expect, vi, beforeEach } from "vitest";
import { getPosts, createPost } from "./posts";
import { supabase } from "./supabaseClient";

// モジュールをモック化
vi.mock("./supabaseClient");

describe("getPosts()", () => {
  beforeEach(() => {
    vi.clearAllMocks();
  });

  it("投稿一覧を取得できる", async () => {
    // モックの戻り値を設定
    vi.mocked(supabase.from).mockReturnValue({
      select: vi.fn().mockResolvedValue({
        data: [
          { id: "1", title: "投稿1", content: "内容1" },
          { id: "2", title: "投稿2", content: "内容2" },
        ],
        error: null,
      }),
    } as any);

    const posts = await getPosts();
    expect(posts).toHaveLength(2);
    expect(posts[0].title).toBe("投稿1");
  });

  it("エラー時は例外をスローする", async () => {
    vi.mocked(supabase.from).mockReturnValue({
      select: vi.fn().mockResolvedValue({
        data: null,
        error: { message: "DB エラー" },
      }),
    } as any);

    await expect(getPosts()).rejects.toThrow("DB エラー");
  });
});
```

### ローカル Supabase に対する結合テスト（モック不使用）

```typescript
// src/lib/posts.integration.test.ts

import { describe, it, expect, beforeAll, afterAll } from "vitest";
import { createClient } from "@supabase/supabase-js";

const supabase = createClient(
  process.env.NEXT_PUBLIC_SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!  // service_role でテストデータを操作
);

describe("Posts CRUD 結合テスト", () => {
  let testPostId: string;

  beforeAll(async () => {
    // テスト用ユーザーを作成
    await supabase.auth.admin.createUser({
      email: "test@example.com",
      password: "password123",
      email_confirm: true,
    });
  });

  afterAll(async () => {
    // テストデータを削除
    await supabase.from("posts").delete().eq("id", testPostId);
    await supabase.auth.admin.deleteUser("test-user-id");
  });

  it("投稿を作成できる", async () => {
    const { data, error } = await supabase.from("posts").insert({
      title: "テスト投稿",
      content: "テスト内容",
      user_id: "test-user-id",
    }).select().single();

    expect(error).toBeNull();
    expect(data.title).toBe("テスト投稿");
    testPostId = data.id;
  });

  it("投稿を取得できる", async () => {
    const { data, error } = await supabase
      .from("posts")
      .select("*")
      .eq("id", testPostId)
      .single();

    expect(error).toBeNull();
    expect(data.title).toBe("テスト投稿");
  });
});
```

## 7. E2E テスト（Playwright）

### セットアップ

```bash
npm install -D @playwright/test
npx playwright install
```

### 認証フローの E2E テスト

```typescript
// e2e/auth.test.ts

import { test, expect } from "@playwright/test";

test.describe("認証フロー", () => {
  test("新規登録 → ログイン → ログアウト", async ({ page }) => {
    // 1. トップページへ
    await page.goto("http://localhost:3000");

    // 2. サインアップページへ
    await page.click("text=新規登録");
    await page.fill('[name="email"]', "e2e@example.com");
    await page.fill('[name="password"]', "password123");
    await page.click('[type="submit"]');

    // 3. ダッシュボードにリダイレクトされる
    await expect(page).toHaveURL(/\/dashboard/);
    await expect(page.locator("text=ようこそ")).toBeVisible();

    // 4. ログアウト
    await page.click("text=ログアウト");
    await expect(page).toHaveURL("/");
  });

  test("ログインエラー時のメッセージ確認", async ({ page }) => {
    await page.goto("http://localhost:3000/login");
    await page.fill('[name="email"]', "wrong@example.com");
    await page.fill('[name="password"]', "wrongpassword");
    await page.click('[type="submit"]');

    await expect(page.locator("text=メールアドレスまたはパスワードが正しくありません")).toBeVisible();
  });
});
```

### RLS の E2E テスト

```typescript
// e2e/rls.test.ts

import { test, expect } from "@playwright/test";
import { createClient } from "@supabase/supabase-js";

// Supabase クライアント（テストデータ準備用）
const supabase = createClient(
  process.env.SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!
);

test.describe("RLS: 他人のデータへのアクセス制限", () => {
  test.beforeAll(async () => {
    // テスト用データを挿入
    await supabase.from("posts").insert([
      { id: "private-post", user_id: "other-user-id", title: "他人の投稿", is_public: false },
    ]);
  });

  test.afterAll(async () => {
    await supabase.from("posts").delete().eq("id", "private-post");
  });

  test("ログインユーザーは他人の非公開投稿にアクセスできない", async ({ page }) => {
    // テストユーザーでログイン
    await page.goto("http://localhost:3000/login");
    await page.fill('[name="email"]', "me@example.com");
    await page.fill('[name="password"]', "password123");
    await page.click('[type="submit"]');

    // 他人の投稿 URL へ直接アクセス
    await page.goto("http://localhost:3000/posts/private-post");

    // 404 または権限エラーページが表示される
    await expect(page.locator("text=見つかりませんでした")).toBeVisible();
  });
});
```

## 8. テストデータ管理

### シードファイルの構成

```sql

  ('11111111-1111-1111-1111-111111111111', 'alice@example.com', crypt('password123', gen_salt('bf')), NOW()),
  ('22222222-2222-2222-2222-222222222222', 'bob@example.com',   crypt('password123', gen_salt('bf')), NOW())
ON CONFLICT DO NOTHING;

INSERT INTO profiles (id, username, avatar_url)
VALUES
  ('11111111-1111-1111-1111-111111111111', 'alice', 'https://example.com/alice.jpg'),
  ('22222222-2222-2222-2222-222222222222', 'bob',   'https://example.com/bob.jpg')
ON CONFLICT DO NOTHING;

-- テスト用投稿
INSERT INTO posts (id, user_id, title, content, is_public)
VALUES
  ('aaaaaaaa-0000-0000-0000-000000000000', '11111111-1111-1111-1111-111111111111', 'Alice の公開投稿', '公開内容', true),
  ('bbbbbbbb-0000-0000-0000-000000000000', '11111111-1111-1111-1111-111111111111', 'Alice の非公開投稿', '非公開内容', false),
  ('cccccccc-0000-0000-0000-000000000000', '22222222-2222-2222-2222-222222222222', 'Bob の投稿', 'Bob の内容', true)

```

## 9. CI での自動実行

### GitHub Actions での全テスト実行

```yaml
# .github/workflows/test-supabase.yml

name: Supabase Test

on:
  push:
    branches: [main, develop]
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - name: Supabase CLI をセットアップ
        uses: supabase/setup-cli@v1
        with:
          version: latest

      - name: ローカル Supabase を起動
        run: supabase start

      - name: pgTAP DB テストを実行
        run: supabase test db

      - name: Node.js をセットアップ
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: "npm"

      - name: 依存関係をインストール
        run: npm ci

      - name: 環境変数を設定
        run: |
          echo "NEXT_PUBLIC_SUPABASE_URL=http://localhost:54321" >> $GITHUB_ENV
          echo "NEXT_PUBLIC_SUPABASE_ANON_KEY=$(supabase status --output json | jq -r '.ANON_KEY')" >> $GITHUB_ENV
          echo "SUPABASE_SERVICE_ROLE_KEY=$(supabase status --output json | jq -r '.SERVICE_ROLE_KEY')" >> $GITHUB_ENV

      - name: Vitest ユニット・結合テストを実行
        run: npm run test

      - name: Supabase を停止
        if: always()
        run: supabase stop
```

## まとめ

| テスト種別 | ツール | コマンド |
|-----------|--------|---------|
| DB ユニット・RLS | pgTAP | `supabase test db` |
| Edge Functions | Deno Test | `deno test --allow-net` |
| フロントエンド結合 | Vitest | `npm run test` |
| E2E | Playwright | `npx playwright test` |

**優先度**: RLS テスト > DB ユニットテスト > 結合テスト > E2E テスト  
RLS のバグはデータ漏洩に直結するため、最優先でテストを整備してください。
