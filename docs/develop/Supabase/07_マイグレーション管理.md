## 1. マイグレーションとは

**マイグレーション**とは、データベースのスキーマ変更を**バージョン管理しながら安全に適用する仕組み**です。

```
問題（マイグレーション無し）:
  開発者 A: ダッシュボードでテーブルを追加 → 本番に反映されない
  開発者 B: 同じテーブルを別の形で作成 → 競合が発生

解決（マイグレーション有り）:
  全員が同じ SQL ファイルを使って変更 → Git で追跡・レビュー可能
```

### マイグレーションファイルの構造

```
supabase/
├── migrations/
│   ├── 20260101000000_create_users.sql
│   ├── 20260115000000_add_posts_table.sql
│   └── 20260228000000_add_index_to_posts.sql
├── seed.sql       ← 初期データ
└── config.toml    ← ローカル設定
```

- **ファイル名**: `{タイムスタンプ}_{説明}.sql`（タイムスタンプ順に適用される）
- **不変の原則**: 一度コミットしたマイグレーションは**絶対に編集しない**。新しいファイルを追加すること。

## 2. Supabase CLI のセットアップ

```bash
# インストール
npm install supabase --save-dev

# バージョン確認
npx supabase --version

# Supabase アカウントにログイン
npx supabase login

# プロジェクトを初期化（初回のみ）
npx supabase init
```

### プロジェクトとのリンク

```bash
# Supabase ダッシュボードのプロジェクト設定 > General からプロジェクト ID を確認
npx supabase link --project-ref <YOUR_PROJECT_ID>

# パスワードを聞かれたら DB パスワードを入力
```

### ローカル環境の起動（Docker 必須）

```bash
npx supabase start
```

起動後に表示される接続情報を `.env.local` に設定します：

```bash
NEXT_PUBLIC_SUPABASE_URL=http://localhost:54321
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJ...（表示された anon key）
```

## 3. マイグレーションファイルの作成

### 方法 1: CLI でファイルを生成する（推奨）

```bash
# マイグレーションファイルを新規作成
npx supabase migration new create_posts_table
```

`supabase/migrations/20260228000000_create_posts_table.sql` が生成されます。このファイルに SQL を記述します：

```sql
-- supabase/migrations/20260228000000_create_posts_table.sql

-- 投稿テーブルの作成
CREATE TABLE public.posts (
  id          UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID        NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  title       TEXT        NOT NULL CHECK (char_length(title) > 0),
  content     TEXT,
  published   BOOLEAN     NOT NULL DEFAULT false,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- インデックス（検索パフォーマンス向上）
CREATE INDEX idx_posts_user_id ON public.posts(user_id);
CREATE INDEX idx_posts_created_at ON public.posts(created_at DESC);

-- updated_at を自動更新するトリガー
CREATE OR REPLACE FUNCTION public.update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER posts_updated_at
  BEFORE UPDATE ON public.posts
  FOR EACH ROW EXECUTE FUNCTION public.update_updated_at();

-- RLS の有効化
ALTER TABLE public.posts ENABLE ROW LEVEL SECURITY;

-- ポリシーの設定
CREATE POLICY "誰でも公開投稿を閲覧できる"
  ON public.posts FOR SELECT
  USING (published = true OR auth.uid() = user_id);

CREATE POLICY "ユーザーは自分の投稿を作成できる"
  ON public.posts FOR INSERT
  WITH CHECK (auth.uid() = user_id);

CREATE POLICY "ユーザーは自分の投稿を更新できる"
  ON public.posts FOR UPDATE
  USING (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);

CREATE POLICY "ユーザーは自分の投稿を削除できる"
  ON public.posts FOR DELETE
  USING (auth.uid() = user_id);
```

### 方法 2: ダッシュボードの変更を取り込む

ダッシュボードで手動変更した内容を SQL として取り込めます：

```bash
# リモート DB との差分を取得してマイグレーションファイルを生成
npx supabase db diff --schema public -f <ファイル名>
```

## 4. ローカル環境での適用

```bash
# ローカル DB にマイグレーションを適用
npx supabase db reset

# ※ db reset はローカル DB を完全にリセットして、
# すべてのマイグレーションを最初から順番に適用する
# シードデータも自動的に投入される
```

### 現在の状態を確認

```bash
# 適用済みマイグレーションの一覧
npx supabase migration list
```

出力例：
```
  LOCAL      │ REMOTE │ TIME (UTC)
  ──────────────────────────────────────────
  20260101000000 │ ✔      │ 2026-01-01 00:00:00
  20260115000000 │ ✔      │ 2026-01-15 00:00:00
  20260228000000 │        │ 2026-02-28 00:00:00  ← まだ本番に適用されていない
```

## 5. 本番環境へのデプロイ

### コマンドによるデプロイ

```bash
# 未適用のマイグレーションを本番 DB に適用
npx supabase db push

# 確認プロンプトなしで適用（CI/CD 向け）
npx supabase db push --password <DB_PASSWORD>
```

> ⚠️ **本番デプロイ前のチェックリスト**:
> - [ ] ローカルで `db reset` して正常に動作することを確認
> - [ ] `migration list` でリモートとの差分を確認
> - [ ] レビュアーが SQL をコードレビュー済み
> - [ ] バックアップが存在することを確認
> - [ ] DROP TABLE / DELETE など破壊的な変更は特に注意

### ロールバック（緊急時）

Supabase は自動ロールバックをサポートしていません。**ロールバック用マイグレーションを手動で作成**します：

```bash
# ロールバック用のマイグレーションを作成
npx supabase migration new rollback_posts_table
```

```sql
-- supabase/migrations/20260228120000_rollback_posts_table.sql
-- posts テーブルのロールバック（緊急時のみ）

DROP TABLE IF EXISTS public.posts CASCADE;
DROP FUNCTION IF EXISTS public.update_updated_at() CASCADE;
```

## 6. シードデータの管理

`supabase/seed.sql` に初期データを定義します。開発・テスト環境でのみ使用します。

```sql
-- supabase/seed.sql

-- テスト用ユーザーデータ（開発環境のみ）
INSERT INTO public.profiles (id, name, bio)
VALUES
  ('00000000-0000-0000-0000-000000000001', '開発者A', 'テスト用アカウント'),
  ('00000000-0000-0000-0000-000000000002', '開発者B', 'テスト用アカウント');

-- テスト用投稿データ
INSERT INTO public.posts (user_id, title, content, published)
VALUES
  ('00000000-0000-0000-0000-000000000001', '初めての投稿', 'これはテストです', true),
  ('00000000-0000-0000-0000-000000000001', '下書き投稿', '公開前の記事', false),
  ('00000000-0000-0000-0000-000000000002', '別ユーザーの投稿', 'テストデータ', true);
```

```bash
# シードデータのみ適用（db reset で自動的に適用される）
npx supabase db reset
```

## 7. CI/CD への統合

### GitHub Actions での自動デプロイ

```yaml
# .github/workflows/database-migration.yml
name: Database Migration

on:
  push:
    branches:
      - main
    paths:
      - 'supabase/migrations/**'
      - 'supabase/seed.sql'

jobs:
  migrate:
    runs-on: ubuntu-latest
    environment: production

    steps:
      - uses: actions/checkout@v4

      - uses: supabase/setup-cli@v1
        with:
          version: latest

      # ローカル Supabase を起動してマイグレーションを検証
      - name: Verify migrations
        run: |
          npx supabase start
          npx supabase db reset
          npx supabase stop

      # 本番 DB にデプロイ
      - name: Deploy to production
        run: |
          npx supabase link --project-ref ${{ secrets.SUPABASE_PROJECT_REF }}
          npx supabase db push --password ${{ secrets.SUPABASE_DB_PASSWORD }}
        env:
          SUPABASE_ACCESS_TOKEN: ${{ secrets.SUPABASE_ACCESS_TOKEN }}
```

### プルリクエスト時のプレビュー環境

```yaml
# .github/workflows/preview.yml
name: Preview Environment

on:
  pull_request:
    types: [opened, synchronize]

jobs:
  preview:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: supabase/setup-cli@v1
        with:
          version: latest

      # マイグレーションの検証（ローカルで実行）
      - name: Validate migrations
        run: |
          npx supabase start
          npx supabase db reset
          echo "マイグレーション検証完了"
          npx supabase stop
```

## 8. よくある操作パターン

### カラムの追加

```sql
-- supabase/migrations/20260301000000_add_tags_to_posts.sql

-- カラムを追加（デフォルト値付き）
ALTER TABLE public.posts
  ADD COLUMN tags TEXT[] NOT NULL DEFAULT '{}';

-- NULL 許容カラムの追加（デフォルト値なし）
ALTER TABLE public.posts
  ADD COLUMN thumbnail_url TEXT;
```

### カラムの変更・リネーム

```sql
-- supabase/migrations/20260302000000_rename_content_to_body.sql

-- カラム名の変更
ALTER TABLE public.posts
  RENAME COLUMN content TO body;

-- カラムの型変更（データが消えない場合のみ）
ALTER TABLE public.posts
  ALTER COLUMN title TYPE VARCHAR(255);
```

### 外部キーと参照整合性

```sql
-- supabase/migrations/20260303000000_create_comments_table.sql

CREATE TABLE public.comments (
  id         UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  post_id    UUID        NOT NULL REFERENCES public.posts(id) ON DELETE CASCADE,
  user_id    UUID        NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  body       TEXT        NOT NULL CHECK (char_length(body) > 0),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- インデックス
CREATE INDEX idx_comments_post_id ON public.comments(post_id);

-- RLS
ALTER TABLE public.comments ENABLE ROW LEVEL SECURITY;

CREATE POLICY "誰でもコメントを閲覧できる"
  ON public.comments FOR SELECT USING (true);

CREATE POLICY "認証ユーザーはコメントを追加できる"
  ON public.comments FOR INSERT
  WITH CHECK (auth.uid() = user_id);
```

### ビューの作成

```sql
-- supabase/migrations/20260304000000_create_post_summary_view.sql

-- 投稿サマリービュー（著者情報を JOIN）
CREATE OR REPLACE VIEW public.post_summaries AS
SELECT
  p.id,
  p.title,
  LEFT(p.body, 200) AS excerpt,  -- 本文の冒頭 200 文字
  p.published,
  p.created_at,
  u.email AS author_email,
  pr.name AS author_name
FROM public.posts p
JOIN auth.users u ON p.user_id = u.id
LEFT JOIN public.profiles pr ON p.user_id = pr.id
WHERE p.published = true;
```

## 9. トラブルシューティング

### マイグレーションが適用されない

```bash
# ローカル環境のリセット（全マイグレーションを再適用）
npx supabase db reset

# マイグレーションの状態確認
npx supabase migration list
```

### 本番とローカルの差分が大きい

```bash
# リモートの現在の状態を取得
npx supabase db pull

# → supabase/migrations/ に差分が生成される
# ただし既存マイグレーションと競合する場合は手動で整理が必要
```

### 「relation does not exist」エラー

マイグレーションの適用順序の問題です。

```sql
-- ❌ 問題: posts より前に comments を作成しようとしている
CREATE TABLE comments (
  post_id UUID REFERENCES posts(id)  -- posts がまだ存在しない
);

-- ✅ 解決: マイグレーションファイルのタイムスタンプを調整して
--         posts を先に作成する
```

### よくある間違い

| 間違い | 正しい方法 |
|--------|-----------|
| 既存のマイグレーションを編集する | **新しいマイグレーションを追加**する |
| ダッシュボードで本番 DB を直接変更する | マイグレーションファイルを作成して `db push` する |
| シードデータを本番に適用する | シードは開発環境専用。本番は別途データ移行スクリプトを使う |
| `db push` を CI/CD テストなしで実行する | 必ずローカル `db reset` で検証してからデプロイ |

## 📌 開発フロー（推奨）

```
1. ブランチ作成
   git checkout -b feature/add-comments

2. マイグレーション作成
   npx supabase migration new add_comments_table

3. SQL を記述して保存

4. ローカルで検証
   npx supabase db reset
   # → アプリで動作確認

5. コミット・プッシュ
   git add supabase/migrations/
   git commit -m "feat(db): add comments table"
   git push

6. プルリクエスト → コードレビュー

7. main にマージ → GitHub Actions が自動デプロイ
```
