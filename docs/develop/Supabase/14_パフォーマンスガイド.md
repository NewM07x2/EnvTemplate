## 1. パフォーマンス問題の全体像

```
よくあるボトルネック:

┌──────────────────────────────────────────┐
│ ① インデックス不足（最も多い原因）        │ → テーブルフルスキャン
├──────────────────────────────────────────┤
│ ② N+1 クエリ                             │ → 大量のリクエスト
├──────────────────────────────────────────┤
│ ③ SELECT * での不要なカラム取得           │ → データ転送量が多い
├──────────────────────────────────────────┤
│ ④ 接続プール不足                          │ → 接続待ちが発生
├──────────────────────────────────────────┤
│ ⑤ RLS ポリシーが複雑すぎる               │ → 毎クエリに認証コスト
└──────────────────────────────────────────┘
```

**改善の優先順位**:
1. スロークエリを特定（計測が先）
2. インデックスを追加
3. クエリを見直す（N+1 解消・SELECT *削除）
4. 接続プールを設定
5. キャッシュを導入

## 2. スロークエリの特定

### Supabase ダッシュボードのログ機能

`Supabase ダッシュボード → Logs → Postgres` で実行されたクエリとその実行時間を確認できます。

### pg_stat_statements で重いクエリを洗い出す

```sql
-- pg_stat_statements を有効化（ダッシュボードの SQL Editor で実行）
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

-- 実行時間が長いクエリ TOP 10
SELECT
  query,
  calls,
  total_exec_time,
  mean_exec_time,
  rows,
  100.0 * shared_blks_hit / NULLIF(shared_blks_hit + shared_blks_read, 0) AS hit_percent
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 10;
```

### EXPLAIN ANALYZE で実行計画を確認する

```sql
-- クエリの実行計画を表示
EXPLAIN ANALYZE
SELECT p.*, u.username
FROM posts p
JOIN profiles u ON p.user_id = u.id
WHERE p.is_public = true
ORDER BY p.created_at DESC
LIMIT 20;
```

出力の読み方：

```
Limit  (cost=... rows=20 ...)
  → Sort  (cost=... rows=...)
      Sort Key: p.created_at DESC
      → Hash Join  (cost=... rows=...)
          →  Seq Scan on posts p  ← ⚠️ フルスキャン！インデックスが必要
               Filter: (is_public = true)
          →  Hash  (cost=...)
               →  Seq Scan on profiles u
```

`Seq Scan`（シーケンシャルスキャン）が出たらインデックスを検討します。

## 3. インデックス戦略

### よく使うインデックスパターン

```sql
-- ① 単一カラムインデックス（WHERE 句で頻繁に使うカラム）
CREATE INDEX CONCURRENTLY idx_posts_user_id ON posts(user_id);
CREATE INDEX CONCURRENTLY idx_posts_created_at ON posts(created_at DESC);

-- ② 複合インデックス（複数条件で絞り込む場合）
-- WHERE is_public = true ORDER BY created_at DESC → is_public + created_at
CREATE INDEX CONCURRENTLY idx_posts_public_created
  ON posts(is_public, created_at DESC)
  WHERE is_public = true;  -- ← 部分インデックス（条件付き）

-- ③ カバリングインデックス（SELECT するカラムをインデックスに含める）
-- → テーブルへのアクセスを不要にする（Index Only Scan）
CREATE INDEX CONCURRENTLY idx_posts_list
  ON posts(is_public, created_at DESC)
  INCLUDE (id, title, user_id);

-- ④ 全文検索インデックス（テキスト検索用）
CREATE INDEX CONCURRENTLY idx_posts_fts
  ON posts USING gin(to_tsvector('japanese', title || ' ' || content));
```

### CONCURRENTLY の重要性

```sql
-- ❌ 本番では危険（テーブルロックが発生）
CREATE INDEX idx_posts_user_id ON posts(user_id);

-- ✅ 本番では必ずこちらを使う（ロックなしで作成）
CREATE INDEX CONCURRENTLY idx_posts_user_id ON posts(user_id);
```

### 不要なインデックスの確認

```sql
-- インデックスが使われていないものを確認
SELECT
  schemaname,
  tablename,
  indexname,
  idx_scan AS scans,  -- 0 なら使われていない
  pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
ORDER BY idx_scan ASC, index_size DESC;
```

## 4. クエリ最適化

### SELECT * を使わない

```typescript
// ❌ 不要なカラムまで取得してしまう
const { data } = await supabase.from("posts").select("*");

// ✅ 必要なカラムだけ取得
const { data } = await supabase
  .from("posts")
  .select("id, title, created_at, user_id");
```

### ページネーション：OFFSET より Range を使う

```typescript
// ❌ OFFSET ページネーション（大量データで遅くなる）
// OFFSET 10000 は 10000 行を読み捨てるので遅い
const { data } = await supabase
  .from("posts")
  .select("*")
  .range(10000, 10019);

// ✅ カーソルベースのページネーション（常に速い）
// 最後に取得した created_at を次のリクエストに渡す
const { data } = await supabase
  .from("posts")
  .select("id, title, created_at")
  .lt("created_at", lastCreatedAt)  // 前回の最後の値より古いものを取得
  .order("created_at", { ascending: false })
  .limit(20);
```

### Postgres 関数でクエリをまとめる

複数クエリをまとめて 1 回の通信にすることでレイテンシを削減できます。

```sql
-- 複数の集計を 1 クエリにまとめた Postgres 関数を作成
CREATE OR REPLACE FUNCTION get_dashboard_stats(p_user_id UUID)
RETURNS JSON
LANGUAGE plpgsql SECURITY DEFINER
AS $$
DECLARE
  result JSON;
BEGIN
  SELECT json_build_object(
    'post_count',    (SELECT COUNT(*) FROM posts WHERE user_id = p_user_id),
    'comment_count', (SELECT COUNT(*) FROM comments c
                      JOIN posts p ON c.post_id = p.id
                      WHERE p.user_id = p_user_id),
    'like_count',    (SELECT COUNT(*) FROM likes l
                      JOIN posts p ON l.post_id = p.id
                      WHERE p.user_id = p_user_id)
  ) INTO result;

  RETURN result;
END;
$$;
```

```typescript
// 3 クエリが 1 回の RPC 呼び出しになる
const { data } = await supabase.rpc("get_dashboard_stats", {
  p_user_id: userId,
});
console.log(data.post_count, data.comment_count, data.like_count);
```

### COUNT(*) の代替（大規模テーブル）

```sql
-- ❌ 大規模テーブルで COUNT(*) は遅い
SELECT COUNT(*) FROM posts;

-- ✅ 概算でよければ pg_stat_user_tables を使う（瞬時に返る）
SELECT n_live_tup AS approximate_count
FROM pg_stat_user_tables
WHERE relname = 'posts';
```

## 5. N+1 問題の解消

### 問題のパターン

```typescript
// ❌ N+1 問題: 投稿ごとにユーザー情報を別クエリで取得してしまう
const { data: posts } = await supabase.from("posts").select("*");

for (const post of posts) {
  const { data: user } = await supabase  // 投稿数分のクエリが走る！
    .from("profiles")
    .select("username")
    .eq("id", post.user_id)
    .single();
  post.author = user;
}
```

### JOIN で解決する

```typescript
// ✅ JOIN で一度に取得
const { data: posts } = await supabase
  .from("posts")
  .select(`
    id,
    title,
    content,
    created_at,
    profiles (
      id,
      username,
      avatar_url
    )
  `);

// data の構造:
// [{ id: "...", title: "...", profiles: { id: "...", username: "alice" } }]
```

### 多段の JOIN

```typescript
// 投稿 → コメント → コメント投稿者 を一度に取得
const { data: posts } = await supabase
  .from("posts")
  .select(`
    id,
    title,
    profiles ( username ),
    comments (
      id,
      content,
      created_at,
      profiles ( username, avatar_url )
    )
  `)
  .order("created_at", { ascending: false })
  .limit(10);
```

## 6. 接続プールの設定（PgBouncer）

サーバーレス環境（Vercel Edge Functions / Next.js API Routes）では、リクエストごとに DB 接続を開閉します。接続数が多すぎると **"too many connections"** エラーが発生します。

### Supabase の接続プール設定

Supabase は **PgBouncer** を内蔵しており、2 種類の接続モードを提供します。

| モード | ポート | 用途 |
|--------|--------|------|
| **Transaction（推奨）** | 6543 | サーバーレス・短時間接続 |
| **Session** | 5432 | 長時間接続（マイグレーション等） |

```bash
# Transaction モードの接続文字列（Supabase ダッシュボード → Database → Connection string）
postgresql://postgres.[PROJECT_REF]:[PASSWORD]@aws-0-ap-northeast-1.pooler.supabase.com:6543/postgres
```

### Next.js での設定例

```typescript
// lib/db.ts（Prisma を使う場合）

import { PrismaClient } from "@prisma/client";

// サーバーレス環境でのシングルトンパターン
const globalForPrisma = globalThis as unknown as {
  prisma: PrismaClient | undefined;
};

export const prisma =
  globalForPrisma.prisma ??
  new PrismaClient({
    log: process.env.NODE_ENV === "development" ? ["query", "error"] : ["error"],
  });

if (process.env.NODE_ENV !== "production") globalForPrisma.prisma = prisma;
```

```env
# .env
# Transaction モード（ポート 6543）を使用
DATABASE_URL="postgresql://postgres.[PROJECT_REF]:[PASSWORD]@aws-0-ap-northeast-1.pooler.supabase.com:6543/postgres?pgbouncer=true"
# マイグレーション時はセッションモード（ポート 5432）を使用
DIRECT_URL="postgresql://postgres.[PROJECT_REF]:[PASSWORD]@aws-0-ap-northeast-1.pooler.supabase.com:5432/postgres"
```

```prisma
// prisma/schema.prisma
datasource db {
  provider  = "postgresql"
  url       = env("DATABASE_URL")
  directUrl = env("DIRECT_URL")  // マイグレーション用
}
```

## 7. キャッシュ戦略

### Next.js の fetch キャッシュ

```typescript
// app/posts/page.tsx（Next.js App Router）

// ① 静的キャッシュ（ビルド時に生成・CDN に配置）
async function getPublicPosts() {
  const { data } = await supabase
    .from("posts")
    .select("id, title, created_at")
    .eq("is_public", true)
    .order("created_at", { ascending: false })
    .limit(10);
  return data;
}
export const revalidate = 60;  // 60 秒ごとに再生成（ISR）

// ② タグベースの再検証（特定のデータが更新されたときだけキャッシュを消す）
async function getPost(id: string) {
  const response = await fetch(
    `${process.env.NEXT_PUBLIC_SUPABASE_URL}/rest/v1/posts?id=eq.${id}`,
    {
      headers: {
        apikey: process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
      },
      next: {
        tags: [`post-${id}`],  // タグを付ける
      },
    }
  );
  return response.json();
}

// 投稿が更新されたときにキャッシュを無効化（Server Action）
import { revalidateTag } from "next/cache";

async function updatePost(id: string, data: PostData) {
  await supabase.from("posts").update(data).eq("id", id);
  revalidateTag(`post-${id}`);  // キャッシュを消す
}
```

### React Query によるクライアントキャッシュ

```typescript
// hooks/usePosts.ts

import { useQuery, useMutation, useQueryClient } from "@tanstack/react-query";
import { supabase } from "@/lib/supabaseClient";

// 投稿一覧を取得・キャッシュ
export function usePosts() {
  return useQuery({
    queryKey: ["posts"],
    queryFn: async () => {
      const { data, error } = await supabase
        .from("posts")
        .select("id, title, created_at, profiles(username)")
        .order("created_at", { ascending: false })
        .limit(20);
      if (error) throw error;
      return data;
    },
    staleTime: 1000 * 60,      // 1 分間はキャッシュを新鮮とみなす
    gcTime: 1000 * 60 * 5,    // 5 分間はメモリに保持
  });
}

// 投稿を作成してキャッシュを更新
export function useCreatePost() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: async (newPost: { title: string; content: string }) => {
      const { data, error } = await supabase.from("posts").insert(newPost).select().single();
      if (error) throw error;
      return data;
    },
    onSuccess: () => {
      // 投稿一覧キャッシュを無効化して再取得
      queryClient.invalidateQueries({ queryKey: ["posts"] });
    },
  });
}
```

## 8. リアルタイム購読の最適化

### 不要な購読を減らす

```typescript
// ❌ テーブル全体を購読（すべての変更イベントを受け取る）
const channel = supabase
  .channel("posts")
  .on("postgres_changes", { event: "*", schema: "public", table: "posts" }, callback)
  .subscribe();

// ✅ 必要なイベント・条件に絞る
const channel = supabase
  .channel(`posts:user:${userId}`)
  .on(
    "postgres_changes",
    {
      event: "INSERT",        // INSERT だけ
      schema: "public",
      table: "posts",
      filter: `is_public=eq.true`,  // 公開投稿だけ
    },
    callback
  )
  .subscribe();
```

### 購読の適切な解除

```typescript
// React でのクリーンアップ
useEffect(() => {
  const channel = supabase
    .channel("posts")
    .on("postgres_changes", { event: "INSERT", schema: "public", table: "posts" }, (payload) => {
      setPosts((prev) => [payload.new, ...prev]);
    })
    .subscribe();

  // コンポーネントのアンマウント時に購読を解除（重要）
  return () => {
    supabase.removeChannel(channel);
  };
}, []);
```

### Realtime のフィルタリング

```typescript
// RLS が有効な場合、認証済みクライアントのみ関連データを受信
const channel = supabase
  .channel("my-room")
  .on(
    "postgres_changes",
    {
      event: "*",
      schema: "public",
      table: "notifications",
      filter: `recipient_id=eq.${userId}`,
    },
    (payload) => console.log(payload)
  )
  .subscribe((status) => {
    if (status === "SUBSCRIBED") {
      console.log("購読開始");
    }
  });
```

## 9. ストレージ最適化

### 画像の最適化（変換機能）

Supabase Storage は URL パラメータで**画像リサイズ・形式変換**をサポートします（Pro プラン以上）。

```typescript
// ① オリジナル URL を取得
const { data } = supabase.storage
  .from("avatars")
  .getPublicUrl("user123/avatar.jpg");

// ② 変換オプションを付与（リサイズ・WebP 変換）
const { data } = supabase.storage
  .from("avatars")
  .getPublicUrl("user123/avatar.jpg", {
    transform: {
      width: 200,
      height: 200,
      resize: "cover",   // cover / contain / fill
      format: "webp",    // jpeg / png / webp
      quality: 80,
    },
  });

// URL 例: .../storage/v1/render/image/public/avatars/user123/avatar.jpg?width=200&height=200&...
```

### アップロード時の圧縮

```typescript
// Next.js でアップロード前にクライアントサイドで圧縮
import imageCompression from "browser-image-compression";

async function uploadAvatar(file: File, userId: string) {
  // 圧縮
  const compressed = await imageCompression(file, {
    maxSizeMB: 0.5,      // 最大 500KB
    maxWidthOrHeight: 800,
    useWebWorker: true,
  });

  // Supabase Storage にアップロード
  const { data, error } = await supabase.storage
    .from("avatars")
    .upload(`${userId}/avatar.jpg`, compressed, {
      contentType: "image/jpeg",
      upsert: true,  // 既存ファイルを上書き
    });

  return data;
}
```

## 10. スケーリング戦略

### 読み取り負荷が高い場合

| 方法 | 効果 | コスト |
|------|------|--------|
| **インデックス追加** | 中〜大 | 無料 |
| **クエリ最適化** | 大 | 無料 |
| **キャッシュ（React Query）** | 大 | 無料 |
| **Postgres の Read Replica** | 大 | Pro プラン |
| **Supabase プランのアップグレード** | 中 | 有料 |

### 書き込み負荷が高い場合

```sql
-- バルクインサートで DB 往復を減らす
-- ❌ 1 件ずつ挿入
for item in items:
  INSERT INTO logs (user_id, action) VALUES (item.user_id, item.action);

-- ✅ まとめて挿入
INSERT INTO logs (user_id, action) VALUES
  ('user1', 'view'),
  ('user2', 'click'),
  ('user3', 'purchase');
```

```typescript
// Supabase JS でバルクインサート
const { data, error } = await supabase
  .from("logs")
  .insert([
    { user_id: "user1", action: "view" },
    { user_id: "user2", action: "click" },
    { user_id: "user3", action: "purchase" },
  ]);
```

## 11. モニタリング

### Supabase ダッシュボードの主要メトリクス

`ダッシュボード → Reports` で確認できる指標：

| メトリクス | 確認場所 | アラートの目安 |
|-----------|---------|--------------|
| DB CPU 使用率 | Reports → Database | 常時 70% 超え |
| DB メモリ使用率 | Reports → Database | 常時 80% 超え |
| API リクエスト数 | Reports → API | 急激な増加 |
| スロークエリ | Logs → Postgres | 平均 100ms 超え |
| 接続数 | Reports → Database | 上限の 80% 超え |

### pg_stat_user_tables でテーブルのヘルスチェック

```sql
-- テーブルの読み書き統計とデッドタプル数
SELECT
  relname AS table_name,
  n_live_tup AS live_rows,
  n_dead_tup AS dead_rows,
  ROUND(n_dead_tup::numeric / NULLIF(n_live_tup + n_dead_tup, 0) * 100, 2) AS dead_ratio,
  last_autovacuum,
  last_autoanalyze
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC;
```

`dead_ratio` が 10% を超えている場合は VACUUM を実行します：

```sql
-- 特定テーブルの VACUUM（デッドタプルを削除）
VACUUM ANALYZE posts;
```

### パフォーマンス改善の効果測定

```sql
-- 改善前後でクエリ実行時間を比較
\timing on

-- 改善前のクエリ
SELECT * FROM posts WHERE is_public = true ORDER BY created_at DESC LIMIT 20;

-- インデックス追加後
CREATE INDEX CONCURRENTLY idx_posts_public_created
  ON posts(is_public, created_at DESC) WHERE is_public = true;

-- 改善後のクエリ（同じクエリ）
SELECT id, title, created_at FROM posts WHERE is_public = true ORDER BY created_at DESC LIMIT 20;
```

## まとめ

| 問題 | 対策 | 優先度 |
|------|------|--------|
| クエリが遅い | `EXPLAIN ANALYZE` でインデックス不足を特定して追加 | 🔴 高 |
| N+1 クエリ | `select()` で JOIN して一度に取得 | 🔴 高 |
| SELECT * の多用 | 必要なカラムだけ指定 | 🟡 中 |
| サーバーレスで接続エラー | PgBouncer（ポート 6543）を使用 | 🔴 高 |
| 同じデータを何度も取得 | React Query・Next.js キャッシュを活用 | 🟡 中 |
| リアルタイム購読が多すぎる | 条件を絞る・不要な購読を解除 | 🟡 中 |
| 画像が重い | Storage の変換機能・アップロード前圧縮 | 🟢 低 |
